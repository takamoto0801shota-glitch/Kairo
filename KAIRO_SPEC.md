# Kairo Spec (最優先仕様)

このドキュメントは「仕様」です。提案ではありません。
実装・修正・ロジック設計は必ず本仕様を最優先で従ってください。
矛盾がある場合は必ず本仕様を優先し、変更が必要な場合はユーザーに確認します。

## 【最重要ルール】まとめの出力回数
- **まとめは同一セッション中に1回しか出力してはいけない。**
- 2回目以降の出力は絶対禁止。実装で複数層のガードを設ける。
- **まとめを1回出した後は、何があってもまとめを再度出さない。** 強制的にまとめ後フェーズに入り、フォロー質問のみ返す。ユーザーが「まとめを出して」「要約して」等と依頼しても再出力禁止。
- **初回メッセージでは必ず質問を返す。** 古い状態（前回のスロット埋まり・まとめ済み）が残っている場合は強制リセットする。
- **まとめ後は2回目のまとめを出さず、フォロー文のみ返す。**

## 1. 目的
- Kairoは「正確に判断するAI」ではなく、ユーザーが次の一歩を踏み出せるよう背中を押す存在。
- 質問は「前進体験」を作るためのもの。納得するまで聞き続けない。
- まとめブロックは必ず最後に出し、欠落を許容しない。

## 2. 口調・立ち位置
- 医者・親・先生の立場にならない。
- 感情に入り込みすぎず、ただし感情を素通りしない。
- 「一緒に整理して判断する存在」として振る舞う。
- 「医療っぽい説明者」ではなく、横で並走する相棒として話す。
- 状態を評価するのではなく、ユーザーの感覚を言葉にして返す。

## 3. 質問ルール（最重要）
- 質問は「前進体験」を作るためのもの（情報収集が主目的ではない）。
- 質問は必ず1回の返答で1つだけ。
- 質問は必ず二択（自由記述は禁止）。
- 二択は「A or B」の1行形式で提示する（箇条書きは禁止）。
- 選択肢は意味のある表現で並べる（低/中/高などの単語は禁止）。
 - ただし最初の痛みスコア質問のみ、1〜10の数値回答を許可する（二択にしない）。
- 質問はテンプレ化しない（同じ文面の繰り返し禁止）。
- 質問フェーズでは**判断・助言は混ぜない**。
- 原因推測／緊急性示唆／行動指示は質問フェーズで禁止。
- 判断をユーザーに委ねる質問は禁止。
  - 「どう思いますか」「どうしますか」は禁止。
  - 「行く/行かない/様子を見る」を選ばせるのは禁止。
  - 「試してみたいことはありますか」など行動選択質問は禁止。

## 3.1 禁止質問パターン
以下の質問・選択肢は使用禁止とする。

- 痛みの強さを主観で聞く質問
  - 例：
    - 軽い／中くらい／強い
    - どのくらい痛いですか？

理由：
- 不安による誇張が起きる
- スコアに正規化できない
- 判定精度を下げる

## 3.2 痛み評価の質問形式（必須）
痛みの評価は必ず「行動可能性・日常への影響」で聞く。
質問文はテンプレ化しないが、内容は行動可能性ベースにする。

## 4. 質問回数の下限・上限
- 最低質問回数は5回。
- 質問の上限は設けない（スロット充填を最優先）。
- まとめに移行する条件は「判断スロットが全て埋まった時点」または「カテゴリ・場合による全質問が完了した時点」のいずれか。
- カテゴリによりスロット数が異なる（PAIN/GI/SKIN=5、INFECTION=6、worsening_trend ありで+1等）。そのカテゴリの全スロットが未充填のまま判定＋まとめへ進むことを禁止する。

## 5. 最後の質問
- Kairoが「最後の質問」と判断した場合、質問文頭は
  「最後に」または「最後の質問です」で開始する。

## 6. 確信度（confidence）の定義
- confidence は「結果」として算出する（判定条件ではない）。
- 算出式：
  - confidence =（埋まった判断スロット数 / 6）× 100
- confidence は UI表現や文言の強さ調整にのみ使用する。
- confidence の数値で判定・まとめ・サマリー表示を制御してはいけない。

## 7. 緊急度判定（危険フラグ優先モデル）

緊急度判定は2段階構造とする。

### Phase1：即時RED条件
判断に使う6スロット（pain_score / 痛みの質 / 発症タイミング / 日常動作影響 / 追加症状 / 原因カテゴリ）について、それぞれのリスクレベルが低（0）・中（1）・高（3）に正規化されているものとする。

- **「高」（スコア3）が2つ以上**ある場合、比率計算を行わず**即時🔴**とする。

### Phase2：重症指数計算
Phase1に該当しない場合のみ、以下の重症指数を計算する（「高」が0個または1個のとき）。

各項目を以下でスコア化：
- 低 = 0
- 中 = 1
- 高 = 3

スロット別のレベル定義（固定）：
- pain_score（数値）
  - 1〜4 = 低
  - 5〜6 = 中
  - 7〜10 = 高
- 日常動作影響
  - 「普通に動ける」= 低
  - 「少しつらいが動ける」= 中
  - 「動けないほどつらい」= 高
- 追加症状
  - 「なし」= 低
  - 「吐き気・めまい」= 中
  - 「しびれ・視界異常」= 高
- その他スロット（痛みの質 / 発症タイミング / 原因カテゴリ）は、質問の選択肢順（1つ目=低、2つ目=中、3つ目=高）で固定する。

自由記述回答の扱い（固定）：
- **緊急度判定時のみ**：選択肢に完全一致しない回答でも、意味が最も近い選択肢のスコアを適用する（マッピング）。それ以外の用途ではマッピングしない。
- **出力は型に合わせる**：自由記述をそのまま出さず、あらかじめの型に合わせて整形する。例：「三日前」→「三日前に症状が始まった」、「少しつらいが動ける」→「日常生活は少しつらいが動ける程度である」。
- **複数事実の箇条書き分割（全スロット共通）**：ユーザーが1つの回答に複数の独立した症状・事実を並べた場合（例: 鼻詰まりと発熱、咳と微熱（37.1））、まとめ・確認文の箇条書きでは**事実ごとに1行**に分ける。例：`・鼻詰まりがある`／`・微熱（37.1）がある` のように、1行に「と」で詰め込まない（Phase1 JSON の `otherSymptoms` 等でも要素を分割する）。

重要度係数：
- pain_score ×1.4
- daily_impact ×1.0
- associated_symptoms ×1.0
- onset ×1.0
- quality ×1.0
- cause ×0.8

重症指数：

severityIndex =
（各スコア × 係数 の合計）÷ 18.6

判定基準：
- 0.65以上 → 🔴
- 0.4〜0.64 → 🟡
- 0.4未満 → 🟢

**「高」がちょうど1つだけの場合**：上記指数で計算し、結果が🟢でも**強制的に🟡**とする（最低でも🟡。指数が🔴のときはそのまま🔴）。

### RED抑制ガード（PAIN / INFECTION・必須）
- `PAIN`（痛み系）および `INFECTION`（発熱感染系）で、経過時間（スロット3）が「さっき」または「数時間前」の場合、**🔴判定を禁止**する。
- **例外（INFECTION のみ）**：痛みの強さ（`pain_score`）が **8 以上**のときは、このガードを**適用しない**（短い経過でも 🔴 になり得る）。
- このガードは Phase1（高リスク2つ以上の即時🔴を含む）/ Phase2（加重指数）の**両方**に適用する（最終判定を🔴にしてはいけない。例外は上記 INFECTION＋痛み8以上のみ）。
- 上記条件に該当する場合（ガード有効時）、最終判定は `🟡` または `🟢` のみ許可する。
- カテゴリ判定は会話開始時に確定したカテゴリ（`triageCategory`）を優先し、後続の文脈混入でカテゴリ判定が揺れても、このガードを維持する。

指数や内部計算はユーザーに表示しない。

## 7.1 6スロット固定（動的充填）
判定に使うスロットは以下の6つで固定する（外部仕様は変更しない）。
1. pain_score（1〜10）
2. 痛みの質
3. 発症タイミング
4. 日常動作影響
5. 追加症状
6. 原因カテゴリ

### 7.1.1 症状カテゴリによる質問差し替え（固定）
症状は以下の4カテゴリに分類する：
- PAIN（痛み系）
- SKIN（皮膚粘膜系）
- INFECTION（発熱感染系）
- GI（消化器系）

カテゴリ判定キーワード（例）：
- PAIN: 頭痛 / 傷
- SKIN: ヒリヒリ / かゆい / 赤い / 発疹 / 唇 / 水ぶくれ / 乾燥 / 口
- INFECTION: 熱 / 発熱 / だるい / 咳 / 喉 / 寒気 / 風邪
- GI: 腹痛 / お腹 / 下痢 / 吐き気 / 胃 / 嘔吐 / 便

質問差し替えルール：
- PAINは従来の質問を維持する。
- SKIN / INFECTION / GI の場合は、スロット4・5・6の質問文と選択肢をカテゴリ定義に差し替える。
- 各質問は3択で提示し、選択肢は低リスク→中リスク→高リスクの順に並べる。
- 自由記述は「最も近い選択肢」にマッピングしてスコア計算に使用する（内部計算は従来どおり）。

#### カテゴリ別 質問テンプレ（固定）

**PAIN・INFECTION 特例（最初は区別できない）：**
- 4問目（スロット4）を付随症状にする。PAIN・INFECTION 共通。
- 質問：「これ以外の症状は他にありますか？」
- **選択肢（PAIN でまだ INFECTION に未確定のとき）**：・咳や鼻詰まりがある、・吐き気がある、・発熱がある（低リスク→中→高の順）
- **選択肢（元から INFECTION 系のとき）**：喉主症状で初めから INFECTION としている場合、または既に `triageCategory` が INFECTION の場合は、**「発熱がある」は出さない**。並びは **・これ以外は特にない、・吐き気がある、・咳や鼻詰まりがある**（低→中→高）。
- **喉が痛いなど、喉的なものが主症状の時**：必ず初めからずっと INFECTION 系とする（4問目を待たずに最初から INFECTION）。
- **実装強制**：`triageCategory === "INFECTION"` **または** 主訴・履歴・痛みスロット回答などの合成テキストで **`isThroatMainSymptom` が真**のときは、付随症状の3択を **必ず**「元から INFECTION 系」版（**「発熱がある」なし**・**これ以外は特にない → 吐き気 → 咳や鼻詰まり**）にする。`category` 引数だけに依存せず **`state` と全文脈**で判定する。
- **付随スロットの自発充填禁止（PAIN/INFECTION）**：`triageCategory` が **PAIN** または **INFECTION** のときは、初回発話の `extractAssociatedSymptoms` 等で **`associated_symptoms` を自発的に埋めない**（主症状語「喉」「熱」「頭痛」などが付随に誤マッチし、4問目がスキップされるのを防ぐ）。訂正意図のあるターンのみ上書き可。
- **喉主症状 INFECTION の緊急度（🟢／🟡）の目安**（他条件が同程度のときの寄せ方）：
  - **経過の方向性（worsening_trend）「発症時より悪化している」** → **最低🟡**（痛みが強く **🔴** になる条件を満たさないときは **🔴** にしない）。
  - **経過「変わらない」「回復に向かっている」** → **🟢** 寄り。
  - **痛み方（worsening・②）「しみる感じ」**（最下段・高リスク）→ **🟡** 寄り。**「ヒリヒリする」**（上段）→ **🟢** 寄り。
- **分岐点（重要）**：この質問で **INFECTION 系へ切り替えるのは「発熱の示唆がある回答」のとき**（選択肢「発熱がある」の選択、または自由記述で発熱・頭が熱い・微熱・体温の数値等が含まれる場合）。**咳・鼻詰まり・吐き気のみでは INFECTION にしない**。
- **複合回答**：「咳や発熱がある」「鼻詰まりと発熱」「咳と37.1」など、**発熱や体温の示唆が1つでも含まれる**ときは INFECTION 系へ進む（咳・鼻だけのときは進まない）。

**PAIN（痛み系）**：スロット5（日常生活への影響）を削除。スロット6を最後の質問とする。
- スロット4: 「これ以外の症状は他にありますか？」（上記の選択肢）
- スロット6: 「何かきっかけで思い当たることはありますか？」（最後の質問。既存ルールに従い「最後に〜」から始める）
  - スマホやパソコンを長時間見た
  - 寝不足や疲れが続いている
  - 強いストレスや緊張があった
  - ※PAINカテゴリ中はこの3択を固定し、他カテゴリ（GI等）のスロット6選択肢へ差し替えない（強制）。

SKIN（皮膚粘膜系）：
- スロット数は5個（スロット6は削除）。
- スロット4（見た目）: 「見た目の変化はありますか？」（上から順にリスク低→中→高）
  - 見た目はほとんど変わらない
  - 赤みや乾燥だけ
  - 水ぶくれ・ただれ・できもの
- スロット5（きっかけ）: 「思い当たるきっかけはありますか？」（最後の質問。「最後に〜」から始める）
  - 特に思い当たらない
  - 紫外線や乾燥が強かった
  - 新しい製品や刺激物を使った

INFECTION（発熱感染系）：喉が痛い等の喉主症状の場合は初めから INFECTION。それ以外は4問目で**発熱の示唆がある回答**（選択肢「発熱がある」、または「咳や発熱」「鼻詰まりと発熱」など**発熱・体温の示唆が含まれる**自由記述）のときに PAIN からこのカテゴリになる（咳・鼻詰まり・吐き気のみではならない）
- スロット4（付随症状）: 「これ以外の症状は他にありますか？」（上記「元から INFECTION 系」のときの3択。PAIN 未確定時は PAIN 側の3択）
- スロット5（体温）: 「体温はどのくらいですか？」
  - 平熱に近い
  - 37度台
  - 38度以上
- スロット6（きっかけ）: 「何かきっかけで思い当たることはありますか？」（上から順にリスク低→中→高）
  - 思い当たらない
  - ストレスや疲労
  - 周りが咳をしていた
- ※主症状（喉の違和感や鼻水等）の質問は削除

GI（消化器系）：
- スロット数は5個（スロット4は削除）。
- スロット5（便や吐き気）: 「便や吐き気はどうですか？」
  - 特に変化はない
  - 下痢がある
  - 吐き気・嘔吐がある
- スロット6（きっかけ）: 「何かきっかけで思い当たることはありますか？」
  - 冷えや寒気がある
  - 便秘
  - 食あたり

カテゴリ運用ルール（強制）：
- **3択の選択肢は、画面上で上から順に selectedIndex 0→1→2 と対応し、リスクは低→中→高（集計スコアも 0→1→2 に整合）とする。**
- カテゴリは会話開始時に確定した `triageCategory` を固定して使う。
- 質問途中で文脈が混ざっても、同一セッション中は質問カテゴリを切り替えない。

### 発話優先原則（必須）
- ユーザー発話にスロット情報が含まれる場合は、そのターンで自動充填する。
- 自動充填されたスロットは対応質問をスキップする。
- 既に埋まったスロットは再質問しない。
- 未充填スロットは優先順で必ず確認する（飛ばさない）。

### 内部管理（必須）
```
slot_status = {
  severity: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  worsening: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  duration: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  impact: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  associated: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  cause_category: { filled: boolean, source: "user_spontaneous" | "question_response" | null }
}
```

### 質問生成ルール（必須）
- 1ターンで質問は最大1つ。
- 対象は未充填スロットのみ。
- 優先順：pain_score → worsening → duration → daily_impact → associated_symptoms → cause_category。
- 充填済みスロットの再質問は禁止。
- 6スロットが全て埋まるまで、未充填スロットへの質問を継続する（強制）。
- 6スロット未充填でまとめへ遷移することを禁止する。

## 7.2 Rest & MC Decision Layer（後段レイヤー）
目的：
- 既存の緊急度判定（Phase1/Phase2）には一切影響させない。
- 医学的判断と社会的対応（MC）を分離して提示する。
- 6問固定仕様は変更しない。

### 6スロット固定（質問は動的）
以下の6スロット自体は固定。質問は「未充填スロットのみ」に対して出す。
1. pain_score（1〜10）
2. 痛みの質
3. 発症タイミング
4. 日常生活への影響
5. 追加症状
6. 原因カテゴリ

### 4問目：日常生活への影響（休息判定専用スロット）
質問文（固定）：
- 「今の動ける感じはどれに近いですか？」

選択肢（固定・順番固定）：
1. 普通に動ける
2. 少しつらいが動ける
3. 動けないほどつらい

### 休息判定（daily_impact のみで決定）
- 「普通に動ける」→ restLevel = NONE
- 「少しつらいが動ける」→ restLevel = LIGHT
- 「動けないほどつらい」→ restLevel = STRONG
- 他スロットは休息判定に使わない。
- 痛みスコア（1問目）を休息判定に使わない。
- 付随症状（5問目）を休息判定に使わない。
- restLevelは後段の Rest & MC Decision Layer 専用変数とする。

### カテゴリ増加後の休息判定ルール（固定）
- `PAIN`：スロット5（日常生活への影響）を削除したため、`restLevel = LIGHT` で固定（🔴のときは休息・MCブロックを出さない）
- `SKIN`：常に `restLevel = NONE`
- `INFECTION`：`@kairo/KAIRO_SPEC.md:184-187`（体温3択）を 1:1 で休息判定に使う
  - 「平熱に近い」→ `NONE`
  - 「37度台」→ `LIGHT`
  - 「38度以上」→ `STRONG`
  - 仕組みは `@kairo/KAIRO_SPEC.md:261-268` と同じ（他スロットは休息判定に使わない）
- `GI`：常に `restLevel = LIGHT`

### MC必要性判定
- restLevel = NONE → mcRecommended = false
- restLevel = LIGHT → mcRecommended = optional
- restLevel = STRONG → mcRecommended = true

### 緊急度 × 休息 の最終行動分岐
- urgency = 🟢
  - NONE：自宅様子見
  - LIGHT：自宅休息（会社規定で必要ならオンライン診療可）
  - STRONG：自宅休息（MCが必要ならオンライン診療推奨）
- urgency = 🟡
  - NONE：市販薬推奨
  - LIGHT：市販薬＋休息（MCが必要ならオンライン診療で取得可能レベル）
  - STRONG：市販薬＋強い休息（MC取得目的ならオンライン診療を第一選択）
  - ※ 🟡では、休息理由だけで対面受診推奨に切り替えない。
- urgency = 🔴
  - 行動：対面受診を優先（受診優先度は変更しない）
  - MC取得は副次目的として扱う
  - 休息・MCブロック（🧾）は表示しない

### 出力原則
- 「医学的判断」と「社会的対応（MC）」を分離して説明する。
- オンライン診療は社会的合理性の選択肢として提示する。
- 医学的緊急度は上書きしない。

### 役割分離（固定）
- 緊急度判定：全6問
- 休息判定：4問目のみ

### 意味定義（固定）
- NONE：勤務可能レベル
- LIGHT：勤務は可能だが休息が望ましい
- STRONG：勤務は困難と判断

※ restLevel は医学的重症度ではなく、機能的制限レベルを表す。

## 8. まとめブロック（必須）
- どんな場合でもまとめブロックは必ず出す。
- まとめブロックの欠落は絶対に許容しない。
- **【最重要】まとめブロックは同一セッション中に一度だけ出力する。2回目以降の出力は絶対禁止。**

### 8.0 まとめ前の確認（必須・🟢🟡🔴共通）

#### 🎯 目的
ユーザーが「これが自分の状態だ」と一瞬で納得できる状態を作る。  
不安を残さず、理解→安心→判断へつなげる。

#### ❗️最重要ルール（ここが今までと違う）
- ここでは“迷わせない”
- 「〜そうです」はOKだが、「念のため」「気をつけて」などの弱い保険ワードは禁止（※🔴まとめ側の「📝 今の状態について」は別構成で、**受診を検討する旨が明確な行動文**で締める）
- 判断は1つだけ出す（揺らさない）
- ユーザーに「考えさせる余地」を作らない

#### 出力構造（固定）

##### ① 導入
`今のところ整理できているのは、`

##### ② 箇条書き
- 6スロットの内容をそのまま使用
- 最大8行
- ユーザー情報は絶対に削らない
- 解釈済みの自然な文章（既存の状態・箇条書きルールに従う）
- **1回答に複数の独立した症状・事実がある場合**は、スロットの区切りに関係なく**症状・測定値ごとに行を分ける**（例：`・鼻詰まりがある` と `・微熱（37.1）がある`）。「・AとBがある」の1行押し込みは避ける。

##### ③ 確認文（2行）
- 1行目：確認の聞き方（プールからランダム）。例：`この理解で合っていますか？`／`大きくずれていないかだけ確認させてください`／`この内容で問題なさそうでしょうか？` など
- 2行目：補足の案内（プールからランダム）。例：`もし補足があれば教えてください`／`抜けていることがあれば遠慮なく教えてください` など
- 不安を残す聞き方は禁止。「修正があれば教えて」のスタンス

#### ❌ 禁止事項（超重要）
- 「念のため気をつけて」などの曖昧ワード（🟡判断の弱い保険表現は禁止）
- 判断を2つ以上出す
- 行動指示（ここではやらない）
- 新しい情報の追加
- LLMによる自由生成の判断文

#### ✅ UXのゴール
このブロックを読んだ瞬間にユーザーが

👉「あ、自分はこの状態なんだ」  
👉「とりあえず大丈夫なんだ」  

と思考停止できる状態を作る

#### フロー・実装メモ（従来ルールの継続）
- 6スロット全て埋まる（またはカテゴリ別の全質問完了）まで、**いきなりまとめを出さず**、まず本確認を出す。
- 確認文はまとめブロック形式ではなく、**普通の質問の形**で出す。
- **表示アニメーション**：確認メッセージはすぐに出さず、上から下へスライドインするアニメーションで表示する。
- **付随症状が最初の発言で言われた場合**：確認文・まとめには記載しない。他のスロットが補充された場合は今の形に合うようにLLMで要約する。

**ユーザー応答の扱い**

- **確認文でユーザーが何かしらの返事をした場合は必ずまとめへ移行する**。
- **追加情報**（自由記述で症状・経過などを補足した場合）→ 箇条書きキャッシュに**必ず**反映する（`confirmationExtraFacts` のマージ・Phase1 再実行・不足追補）。まとめは**先行生成済みで全ブロックが揃っている**ときは「今の状態について」**ブロックのみ**差し替え（他ブロックはそのまま）。未生成・未取得のときはまとめを**フル生成**する。クエリ検索時も同じ箇条書きを使う。
- **それ以外**（「あっている」「大丈夫です」「はい」等の肯定のみ）→ **絶対に箇条書きには書かず**、まとめを出す。
- **違う／間違っている**→ まとめを必ず出す（中間メッセージは出さない）。
- **確認文の後にユーザーが何かしら言ったら、まとめをいつでも出してよい。必ずまとめを出す。それ以外のもの（中間メッセージ等）は強制的に出さない。**
- **ユーザーの言葉の仕分け**：肯定のみ（あっている・大丈夫・ない・特にない・はい等）→ 既存まとめを使用。追加情報（症状・経過等を補足）や違う等→ 「今の状態について」ブロックのみ差し替え（他ブロックはそのまま）。
- この順番を必ず守る：確認 → ユーザー応答 → まとめ。
- **確認文は出すだけ**。条件を満たした時点で**確認文の HTTP 応答を先に返す**（Phase1 箇条書きの完了を待たない。確認文の箇条書きはレガシー／簡易生成でよい）。**応答を返した直後**にバックグラウンドで Phase1 箇条書き整形→まとめ本文生成を**直ちに**開始する。
- **カテゴリ別の質問数**：PAIN/GI/SKIN=5スロット、INFECTION=6スロット、worsening_trend ありで+1等、カテゴリと状態によりスロット数が異なる。そのカテゴリの全質問を終了したら、またはスロットが全て埋まったら、確認文へ移行する。
- **全スロット完了時、または質問完了時**：上記のとおり確認文を**先に表示**し、**表示直後**にバックグラウンドでまとめ生成を開始。ユーザー応答時は生成完了を待ってからまとめを表示する。
- ユーザーが応答したら、**まとめを出すだけ**。中間メッセージ・閉じメッセージ・LLM応答等は一切出さない。
- まとめの出力は**ユーザーが確認文に対して何かを言った後**のみ行う。それまでは待機。
- ユーザー応答時、まだまとめ生成が終わっていなければ**完了まで待ってから出力**する。
- **サマリーカードと共にまとめブロックを順に生成**する（既存のルールのまま）。
- **フォロー文はまとめを全て出し終えてから出力する**（どんな場合でも）。
- **まとめの後にはフォロー文を絶対表示する**。
- まとめブロック内に**質問文を一切書かない**（疑問文・?・？は禁止）。
- まとめブロックの後は**フォロー質問フェーズ**に必ず移行する（会話終了にしない）。
- まとめ内容はテンプレ禁止。会話内容に即して毎回生成。

### 8.0.1 まとめを出すルール一覧（実装・仕様全体の参照用）

本セクションは、まとめの出力に関するルールを一箇所に集約した参照用ドキュメントである。

#### A. まとめ出力のトリガー条件（いつまとめに移行するか）

| 条件 | 説明 |
|------|------|
| **全スロット埋まり** | カテゴリ・状態に応じた全スロットが埋まった時点 |
| **カテゴリ別の全質問完了** | そのカテゴリの全質問（スロット数分）を終了した時点 |

**判定式（実装）**:
- `decisionAllowed` = 埋まったスロット数 >= そのカテゴリの必須スロット数
- `questionsCompleted` = 質問回数 >= そのカテゴリの必須スロット数
- `shouldJudgeNow` = `canShowSummary` AND (`decisionAllowed` OR `questionsCompleted`)

**追加の必須条件**:
- `canShowSummary` = 初回ユーザーターンではない AND ユーザーターン数 >= 2
- 初回メッセージでは絶対にまとめを出さない

#### B. カテゴリ別スロット数（必須質問数）

| カテゴリ | スロット数 | 備考 |
|---------|-----------|------|
| PAIN | 5 | daily_impact なし |
| GI | 5 | daily_impact なし |
| SKIN | 5 | cause_category なし |
| INFECTION | 6 | 体温スロットあり |
| 共通 | +1 | duration が「さっき」以外のとき worsening_trend を追加 |

#### C. まとめ出力のフロー（順序厳守）

1. **確認文の表示**：全スロット埋まり or 全質問完了 → いきなりまとめを出さず、まず確認文を出す
2. **まとめ生成の開始**：確認文表示と同時にバックグラウンドでまとめ生成を開始
3. **ユーザー応答の待機**：ユーザーが確認文に何か返答するまで待機
4. **まとめの表示**：ユーザー応答後、生成完了を待ってからまとめを表示
5. **フォロー文の表示**：まとめを全て出し終えてからフォロー文を表示

#### D. 確認文へのユーザー応答の扱い

| 応答タイプ | 処理 |
|-----------|------|
| **肯定のみ**（あっている・大丈夫・ない・特にない・はい等） | 既存まとめを使用。箇条書きには書かず、まとめを出す |
| **追加情報**（症状・経過等を補足） | 「今の状態について」ブロックのみ差し替え。他ブロックはそのまま |
| **違う／間違っている** | まとめを必ず出す（中間メッセージは出さない） |

- 確認文の後にユーザーが何かしら言ったら、**必ずまとめを出す**
- 中間メッセージ・閉じメッセージ・LLM応答等は一切出さない

#### E. まとめの出力回数（最重要）

- **まとめは同一セッション中に1回しか出力してはいけない**
- 2回目以降の出力は絶対禁止
- まとめを1回出した後は、何があってもまとめを再度出さない
- ユーザーが「まとめを出して」「要約して」等と依頼しても再出力禁止
- まとめ後は2回目のまとめを出さず、フォロー文のみ返す

#### F. まとめ出力の禁止条件

- 初回メッセージではまとめを出さない（必ず質問を返す）
- ユーザーターン数が2未満ではまとめを出さない
- 全スロット未充填のまま判定＋まとめへ進むことを禁止
- 確認文の前にまとめを出さない（確認 → ユーザー応答 → まとめの順序厳守）

#### G. エラー時のフォールバック（実装）

- 確認応答時のまとめ生成でエラーが発生した場合、`buildLocalSummaryFallback` でまとめを返す
- API全体の catch で、`wasWaitingForSummary`（確認文表示済み＋まとめ未表示）の場合は、`buildLocalSummaryFallback` でフォールバックまとめを返す
- LLM 呼び出しは失敗時は最大10回までリトライ

#### H. まとめ後のフェーズ

- まとめブロックの後は**フォロー質問フェーズ**に必ず移行する（会話終了にしない）
- まとめ直後に**フォロー質問を1つ**必ず表示する
- まとめ後は2回目のまとめを出さず、フォロー文のみ返す

必ず以下の5ブロックを順に出す（🚨 もし次の症状が出たら は絶対に出さない）：
1. 🟢 ここまでの情報を整理します
2. 🤝 今の状態について

#### 🟢/🟡 ここまでの情報を整理します（ブロック1）

**適用範囲**：様子見/市販薬の場合（B）のまとめブロック先頭。🟢または🟡の緊急度に応じて見出しの絵文字を変える。

**構成**：
- 見出し：`🟢 ここまでの情報を整理します`（🟡の場合は`🟡 ここまでの情報を整理します`）
- 直下に導入文を1文のみ入れる（固定テンプレからランダムで1つ）

**導入文テンプレ（固定）**：
- 「教えてもらった内容をもとに、今の状態を一度まとめますね。」
- 「ここまでに聞いたことを整理して、今の状況を確認しますね。」

**強制ルール**：本文は固定テンプレのみ。LLM生成・その他の自由文は禁止。実装で必ず `enforceSummaryIntroTemplate` を適用し、固定文以外を差し替える。

**ブロック1の位置**：本文は必ず**先頭**（`🤝` より前）。欠落時は付与、中間にあった場合は先頭へ移動。`ensureGreenHeaderForYellow` が病院フロー（`🏥` 等）以外の 🟢/🟡 まとめで必ず適用する。

**禁止事項**：判断・安心・結論を出さない。整理する宣言のみに限定する。

3. ✅ 今すぐやること
4. ⏳ 今後の見通し
5. 🌱 最後に

#### 🤝 今の状態について（🟢/🟡/🔴・確認文共通）

**適用範囲**：🟢/🟡のまとめブロック、🔴の📝今の状態について、まとめ前の確認文。いずれも同じ設計で生成する。

**確認文とまとめの出力形式の違い**（最大6項目）：

- **確認文**：
```
今の情報から見ると、

・〇〇
・〇〇
・〇〇
（最大6項目）

という状況です。
```

- **まとめの「今の状態について」**：箇条書きのみ
```
・〇〇
・〇〇
・〇〇
（最大6項目）
```
※「今の情報から見ると、」「という状況です。」は出さない

**目的**：ユーザーが「なるほど、自分はこういう状態なんだ」と一発で理解できること。転記ではなく**意味理解→文章化**を強制する。

---

**【最重要：箇条書き生成エンジン（2フェーズ強制）】**

**絶対原則**：ユーザー入力のコピー禁止／テンプレ当てはめ禁止／**JSON生成なしの出力禁止**

**Phase1**：意味構造JSON（内部のみ）。これを経由せず箇条書きを出すことは禁止。

**全カテゴリ共通JSON**（キーはこれのみ）：

```json
{
  "main_symptom": "",
  "details": [],
  "onset": "",
  "trend": "",
  "otherSymptoms": [],
  "noOtherSymptoms": false,
  "cause": "",
  "context": ""
}
```

- `details`：補足用。`main_symptom` と意味が重複する内容は入れない（PAINでは痛みを**1文に統合**。「〜ような痛みが出ているような痛み」等の冗長禁止）。
- カテゴリ別の解釈はプロンプトで補強（PAIN／INFECTION／SKIN／GI）。

**自由記述の分解順（必須）**：1 状態 → 2 時間 → 3 強さ・頻度 → 4 原因（推定）

**例**：
- 「太陽をかなり浴びた」→ `"cause": "強い日差しや暑さによる影響の可能性"`
- 「昨日ほぼ寝てなくてスマホ見てた」→ `"cause": "睡眠不足や画面の見過ぎによる負担の可能性"`

**暴走防止**：未言及症状の追加禁止／原因は1つにまとめる／不明は「不明」／病名・断定禁止／「可能性」で止める

---

**Phase2**：箇条書き生成

- **JSONのみ**を参照（ユーザー入力参照禁止）
- 1行＝1意味、**最大6行**、統合優先、自然な日本語
- 実装：`formatBulletsFromMeaningJson`（`main_symptom` を主症状の1行に。`details` は main と重複させないため箇条書きには展開しない）
- **経過時間（onset）・悪化傾向（trend）**：ユーザーが「ない」「わからない」等と答えた場合や、仕様上の**特定の表示専用**を除き、**スロットに実回答がある限り箇条書きに必ず反映する**（欠けた場合は `injectMissingSlotBulletsFromState` でスロットから補完）。表現は**軽く書き言葉に整えてよい**（例：「ちょっと悪化しているかも」→「やや悪化している」「やや発症時より悪くなっている」など。**シンプルな短さでよい**）。緊急度判定の強さに合わせて文言を強める必要はない（表示はユーザーに近い事実優先）。

**正しい出力例**：
```
・やや強いズキズキする頭痛が出ている
・症状は急に始まっている
・他の症状は今のところ見られていない
・強い日差しや暑さによる影響の可能性
```

**NG例**：`・やや強いズキズキする痛みが出ているような痛みが出ている`（JSON未経由・冗長）

**品質チェック（通らなければ再生成）**：同じ意味の繰り返しなし／「〜ような痛み」の重ね掛けなし／1行で意味完結／人間が読んで自然

**再生成条件**：日本語不自然／同語繰り返し／テンプレ感／原因が未解釈のまま

**絶対禁止**：ユーザー文コピー／テンプレ表の直接使用／JSONスキップ／「〜ような痛みが出ているような」構文

---

**■確認応答時の扱い（必須）**

確認文でユーザーが何かしら（大丈夫・はいなどの肯定、または追加情報）言った場合、まとめの箇条書きにも**同じ2段階生成**で対応する。

- 肯定のみ（大丈夫、はい、合っている等）→ 既存の意味JSONベースの箇条書きをそのまま使用
- 追加情報あり（あと熱も出た、吐き気も少しある等）→ その内容を生入力に含め、2段階生成で**新たに意味JSON→箇条書き**を生成。絶対にそのまま出したり、変換せずにテンプレで出したりしない
- 否定のみ（違う等、情報がない否定）→ 箇条書きには追加しない

**例**（確認時に「あと熱が出た」と返答した場合の箇条書き）：
```
・やや強いズキズキする頭痛が出ている
・症状は急に始まっている
・発熱を伴っている
・他の症状は今のところ見られていない
・睡眠不足や画面の見過ぎによる負担の可能性
```
※「あと熱が出た」をそのまま並べず、意味に変換して統合した形で出力する。

**何があっても**：ユーザー発話をそのまま表示しない。テンプレ当てはめで済ませない。

---

**絶対ルール**：
- ユーザーの文を第一優先とする。ユーザーが言ったことのみを使う。
- 「ない / わからない / 不明 / 思い当たらない」等の回答は表示しない。
- 複数要素（例：太陽光と睡眠不足）が1回答に含まれる場合は、まとめ行でも必ず優先して保持する。
- 箇条書きの後に改行を挟む
- 構成：事実 → 判断（感想・解説は禁止）
- 診断・断定・医療判断は禁止
- 「だから」「なので」等の接続語で判断を明示
- 「〜かもしれません」は最大1回まで
- 行動指示禁止

#### 🤝 今の状態について（🟢/🟡・完全置き換え）
🟢・🟡では**従来の共感テンプレ＋「ただ、」＋判断1文は廃止**し、次の **3 ブロック固定構成**とする（**LLM 禁止・サーバ固定**）。**🟢 と 🟡 で本文プールは同一**（緊急度の差はまとめ見出しや他ブロック側で表現）。
##### ① 状態の定義（事実の組み合わせ）
- 1 行目：**`「〇〇」＋「〇〇」（必要なら＋「〇〇」）`状態です。**（**最低 2 つ・上限 3 つ**の「」を **＋** でつなぐ。）
- **〇〇** の取り方は 🔴 の「組み合わせ行」（上記 **7.1.1 付近・HIGH 列挙**）と**同じスロット由来の短い名詞句**だが、ここでは緊急度が **LOW〜MEDIUM のスロットだけ**を対象にする（**HIGH は含めない**）。**主症状ラベルそのものは 〇〇 に含めない**（🔴と同様）。**箇条書きから取る場合も** `slotNormalized` で **LOW〜MEDIUM の行に対応**するものだけ**とし、**主症状の説明行**（通常は痛みの強さ行の次の1行）は**除外**する。
- **要約の出どころ**：上記の理想形に加え、**まとめの「今の状態について」箇条書き**（`stateAboutBulletsCache` / `buildStateFactsBullets` が生成する行）と**同じ要約パイプライン**から **〇〇 を組み立ててよい**（ユーザーに見える内容の整合が取りやすい）。**再スロット走査より速く出せるなら、実装は箇条書きキャッシュを優先してよい**（LOW〜MEDIUM の制約・主症状除外は従来どおり満たすこと）。
- **短い語への要約**：箇条書きの1行をそのまま使わず、**組み合わせ行用にさらに短く**する（例：`・ズキズキする痛み` → `ズキズキ`、`やや強いズキズキする頭痛` → `やや強い頭痛` のように**痛み方の語を落とす**）。
- **逆優先度（なるべく出さない）**：**きっかけ**（`cause_category` 由来）と**痛み方**（`worsening`・痛みの質）の要素は、**他に候補が足りるときは選ばない**（付随・体温・経過・悪化傾向・日常生活への影響・痛みの強さなどを優先）。**最低2・上限3**を満たすために必要なときだけ繰り上げる。
- **要素数は最低 2 つ・上限 3 つ**（1 つしか取れないときは、サーバで許容される短い補助語を足して 2 つにする。4 つ以上取れる場合は優先度の高い順に 3 つまで）。
##### ② 状態の意味（パターン化）
- 続けて：**`この組み合わせは、`** と **`一時的な〇〇としてよく見られるパターンです。`**（改行は可だが本文中に**空行は入れない**）

- **例外（`PAIN` / `INFECTION` のみ・サーバ固定）**：次のいずれかに当てはまるときは、上記の「一時的な〇〇として〜」**の代わりに** **`風の初期症状としてよく見られるパターンです。`** を**そのまま**用いる（`〇〇` は使わない）。

##### 🤝/📝 LLM安全フィルタ仕様
- 出力フォーマット固定：`{ "common": [], "conditional": [], "rare_emergency": [] }`
- **common**：最低2件・最大4件。**conditional**：最低2件・最大4件。**rare_emergency**：**2件のみ**（強制）。
- **common** = 一般的に頻度が高い原因。`・<原因名> → <理由>` の理由は**ユーザーの言動を要約**して記載。固定文にしない。**主症状と無関係な病名・症状名は絶対に含めない**。無理に病名を挿入しない。
- **conditional** = 条件付きで考慮。`・<病名> → <関連した理由>` 形式。**主症状に関連する病名のみ**使用。主症状と無関係な病名は絶対に含めない。理由はユーザー症状と関連付ける。検索結果＋症状まとめから要約。固定テンプレート禁止。
- **キーワード重複禁止**：一度出たキーワード（ズキズキ・痛み・頭痛 等）は、同じ箇条書き内で繰り返さない。
- **rare_emergency** = 稀だが緊急性あり（デフォルト非表示）。**2件のみ**（強制）。検索結果＋症状まとめから要約。固定テンプレート禁止。

1. 状況の自然な流れを一言で述べる（断定しない）

   ・「もし痛みが7以上に強くなったら」

3. 必ず次の一文で締める（固定表現）
   「そのタイミングで、もう一度Kairoに聞いてください。」

トーン：
- 落ち着いている
- 先回りして隣にいる感じ
- 不安を増やさず、行動の逃げ道を用意する

## 15. 位置情報と受診先の具体提示（必須）
- UI表示・会話生成・判断ロジックを完全に分離する。
- 位置情報は「LocationSnapshot」として一度だけ確定保存する。
- Snapshot はセッション終了まで上書き・初期化・破棄しない。
- 位置情報が取れなくても会話は止めない。
- 「📍現在地取得済み」は lat/lng のみで判定する。
- ユーザーに現在地をテキスト入力で聞かない。
- 失敗理由をユーザーの責任にしない。

### 15.1 LocationSnapshot（必須）
```
type LocationSnapshot = {
  lat: number;
  lng: number;
  ts?: number;
};
```

### 15.2 正規化関数（必須）
すべての生locationは必ずこの関数を通す。
```
function normalizeLocation(raw: any): LocationSnapshot | null {
  if (!raw) return null;
  if (raw.lat != null && raw.lng != null) {
    return { lat: raw.lat, lng: raw.lng, ts: raw.ts };
  }
  return null;
}
```

### 15.3 conversationState 初期化（必須）
```
function initConversationState(input?: Partial<State>): State {
  return {
    conversationId: input?.conversationId ?? generateId(),
    locationSnapshot: input?.locationSnapshot ?? null,
    clientMeta: input?.clientMeta ?? {},
  };
}
```
- 未初期化代入は禁止。必ず factory 経由で生成する。
- `state.location.xxx = ...` の直接代入は禁止（全体置換のみ）。

### 15.4 Snapshot の確定条件（厳守）
- lat と lng が取得できた時点で snapshot を確定する。
- city / reverse geocode / timestamp は一切条件に含めない。

### 15.5 取得フロー（重要）
- 初回ロード時に自動で取得を開始する。
- lat/lng が取得できた時点で snapshot を確定する。
- snapshot が無い限りは「未取得」扱いを維持する。

### 15.6 UI表示ルール（厳守）
- lat/lng がある場合：
  - 「📍現在地取得済み」を表示する。
- lat/lng がない場合：
  - 「📍現在地を取得できていません」を表示する。

### 15.7 位置情報取得UI（必須）
- 初回のみ以下の文言を必ず一度だけ表示（改変禁止）：
```
より正確な案内のため、現在地を使用できます。
今回は許可しなくても会話は続けられます。
```
- 会話は止めない。
- 初回のみ permission UI を出し、その後は裏で retry。

### 15.8 フォールバック設計
- どの状態でも会話は通常進行。
- snapshot がない場合のみ案内を停止する。

### 15.9 判断ロジック（最重要）
```
function canRecommendSpecificPlace(snapshot: LocationSnapshot | null) {
  return snapshot?.lat != null && snapshot?.lng != null;
}
```
- snapshot がある場合のみ → 具体名OK

### 15.10 医療施設案内（city 非依存・強制案内）
- 施設案内の判断基準は lat / lng のみ。
- snapshot（lat/lng あり）なら必ず案内する。
- city / country / locality は判断ロジックに使わない（表示のみ）。
- 検索範囲は半径 500m〜1km。
- 対象カテゴリ（症状に合わせて選ぶ。GP固定は禁止）：
  - 歯の痛み：歯医者（dentist / dental clinic）
  - 耳・鼻・のど：耳鼻科（ENT / ENT clinic）
  - それ以外：GP（clinic / general practitioner / medical clinic）
  - 重症・緊急が疑われる場合：病院（hospital / medical centre）
  - 薬局（SG）：pharmacy / Watsons / Guardian
- 表示方法：
  - 「近くで行きやすい場所を案内します」を必ず入れる
  - 候補は最大3件、距離が近い順、1件目は「おすすめ」
  - Google Maps をワンタップで開けるリンクを必ず付ける（lat/lng + query）
- city 未取得を理由に案内中止しない
- 「現在地が取得できていないため〜」等の文言は禁止

### 15.10 施設案内の検索ルール（必須）

### 🇸🇬 シンガポール専用ロジック

- **戦略**：日本人クリニック OR GP のみ。専門科検索禁止。
- **① 日系**：Text Search `japanese clinic singapore`、半径制限なし、max3件
- **② GP**：Nearby Search、**radius=1000（1.0km以内に限定）**、type=doctor、keyword=clinic/GP/family medicine
- **③ 除外**：aesthetic, tcm, physio, animal, veterinary, pediatric（小児科）。**歯痛時は dental を検索・表示**。
- **④ スコア**：`japaneseClinic*3 + gpMatch*3 + distanceScore*2 + ratingScore`（distance: 0-500m=2, 500-1k=1, 1k超=-1。rating 3.8未満は除外）
- **⑤ 意思決定**：**主役＝GP（必須）**。`main=GP1`、`alternatives=[日系(あれば), GP2, 日系orGP]`
- 出力：`candidates = [GP, 日系?, GP?, 日系?]`（本文2件、モーダル最大5件）

**目的**：保険の不安を解消しつつ、誤情報を出さない。Google Placesからは保険情報を取得できないため、施設タイプから「傾向ベース」で表示する。

**例**：`① Raffles Medical（ラッフルズ・メディカル）［保険: ◯ 使えることが多い］`

### 8.1 Places API 実装詳細（現行仕組み）

#### 環境変数
- `GOOGLE_PLACES_API_KEY` または `GOOGLE_MAPS_API_KEY` を指定
- 未設定時は `[]` を返し、検索をスキップ

#### 使用するAPI（既存実装の4種のみ使用）
1. **Nearby Search** (`nearbysearch`)：`location`, `radius` or `rankby=distance`, `keyword`, `type`
2. **Text Search** (`textsearch`)：`query`, `location`, `radius`, `type`（省略可）
3. **Place Details** (`place/details`)：`place_id`, `fields`: rating, reviews, types, url, editorial_summary
4. **Geocoding** (`geocode`)：住所↔緯度経度、逆ジオコード

#### 医療機関検索フロー（`fetchCarePlacesWithFallbacks`）
1. **Nearby Search（半径順）**
   - types: `doctor`, `hospital`, `health`
   - 半径: 1km → 3km → 5km → 10km → 20km → 50km（結果が出るまで順に試行）
   - 日本: keywords に `クリニック`, `内科`, `病院`, `医療`, `診療所` を追加
2. **rankByDistance（距離順）**
   - 0件の場合、`rankby=distance` で type ごとに検索（半径制限なし）
3. **Text Search（type 指定あり）**
   - 0件の場合、`type=doctor` で radius 5km〜50km を順に試行
4. **Text Search（type 指定なし）**
   - 0件の場合、`type` なしで検索（より広い結果）
5. **都市名検索**
   - 0件の場合、逆ジオコードで都市を取得し、`[都市名] 病院 クリニック` で検索
6. **0件ログ**
   - キー・位置が有効なのに全戦略で0件の場合、`console.error` でログ出力

#### 候補取得フロー（`resolveCareCandidates`）
1. 位置取得: `locationSnapshot` → ジオコード（住所）→ `getFallbackCoordinates`
2. **国判定**: lat/lng から逆ジオコードで `country` を取得。`country === "Singapore"` の場合はシンガポール専用ロジックへ
3. **シンガポールの場合**（`fetchCarePlacesForSingapore`）：🇸🇬 シンガポール専用ロジックに従う。
   - ① 日本人クリニック: Text Search `japanese clinic singapore`、半径制限なし、最大3件
   - ② GP: Nearby Search、radius=1000、type=doctor、keyword=GP/clinic/family medicine
   - ③ 除外: aesthetic, tcm, physio, animal, pediatric（小児科）等。歯痛時は歯科を検索・表示。
   - ④ スコアリング後、⑤ 意思決定：hasJapanese ? main=日系・alternatives=GP2件 : main=GP1・alternatives=GP2件
   - 出力: `[main, alt1, alt2]`（最大3件）
4. **他国**：`detectCareMainSymptomText` → `buildCareSearchQueries` → `fetchCarePlacesWithFallbacks`
5. `mergePlaces` で重複除去 → 他国のみ `applySymptomFitFilter` → `prioritizeCareCandidates`（SGは順序維持）
6. Place Details で `rating`/`reviews`/`editorial_summary` 付与。SGは最大3件、他国は最大2件。

#### いいところ生成（`buildHospitalRecommendationReasons`）
優先順：① 症状との相性（`getSymptomFitReasonsByCategory`）② 日本語対応 ③ 通いやすさ ④ 初診の相談先。`usedReasons` で重複回避。禁止：評判が良い、人気、おすすめ等。

#### 病院・クリニック・薬局
- `resolveHospitalCandidates`：SG→`[]`。他国→hospital/doctor、rankByDistance、5〜50km。
- `resolveClinicCandidates`：doctor/hospital、クリニック/内科（日本）、5〜20km。
- `resolvePharmacyCandidates`：pharmacy、薬局/ドラッグストア（日本）、5〜50km。

#### フォールバック
- フォールバック候補（固定名称）は使用しない
- API 結果が 0 件の場合は `candidates: []` を返し、「位置情報から検索しましたが、見つかりませんでした」と表示

## 9. 「🤝 今の状態について」の出力順
順番を厳守：
1. ユーザーのつらさ・不安への一文の寄り添い
2. ユーザーの事実要約（箇条書き、改行）
3. ユーザーの感覚を言葉にして返す（感覚の翻訳）
4. Kairoとしての判断（断定しすぎない）

追加ルール：
- ③は「今の話を聞く限りだと」など、ユーザーの話が主語になる形にする。
- ③は一般論の説明ではなく、感覚の翻訳を優先する（医者っぽさ禁止）。
- ③で「一般的には〜」など医療説明を主にしない（💊以外で禁止）。
- 診断・確定表現禁止。原因の断定禁止。
- 「注意が必要です」は禁止。代わりに「今すぐ慌てる感じではなさそうです」等を使う。
- 3〜5文程度に収める。
- 「今のあなたの状態なら、こう考えて大丈夫です」「だから今日はこれでいいですよ」は方向性の例。
- テンプレ化はせず、ユーザーを落ち着かせる表現で言い換える。

## 10. 「✅ 今すぐやること」の生成ルール

### 目的
「今すぐやること」は、単なる行動提示ではなく、**納得 → 行動 → 安心**の順で心理的不安を下げる設計にする。
- ユーザーが読んだ後に：今やることが明確／それで良い理由が理解できる／どのくらい続くか想像できる／病院に行かなくてもいい可能性を自然に感じられる、という状態を作る。

### 10.1 出力構造（必ずこの順番）

#### ① なぜそれでいいのか（安心の土台・2文以内）
- 現在の状態は一時的な変化である可能性があるという一般的説明
- 今の段階では刺激を減らすことが合理的である、という理由提示
- 診断・断定禁止。「かもしれません」は最大1回

#### ② 今すぐやること（具体・最大3項目）
- **モーダルと同じクエリ**（10.3：主症状 + 箇条書き全文 + 対処法）を使用して検索し、検索から**必ず2件**を出す（特例の固定項目はこの2件に含めない）。
- **箇条書きの文の長さ・簡潔さ・構成はモーダル（✅ 具体的に）と同一**とする。行動文は簡潔に（長い場合は適宜省略）。理由文も簡潔に。参考表示（参考:...）はモーダルと同様に省略する。
- **主症状・ユーザーの回答を「付随症状」にまとめない**。主症状は主症状として、各スロットの内容を適切に区別して行動・理由に反映する。
- **口調**：勧める口調を使う（`〜してください` / `〜するといいです`）。`・〜します` のような一人称は避ける。
- **PAIN系・INFECTION系で🟡のとき**：1件目を以下で固定する（特例。この固定の次に検索2件を追加）。**本文の ✅ ブロックにのみ適用**（`✅` モーダルの `■今すぐやること` には含めない）。
  - `・今はベッドに入り、横になって数時間ゆっくり過ごしてください`
  - `→ 体を休息モードに切り替えることで、自然な回復の流れが働きやすくなります。`
- 各件は必ず次の2行セットで出力する。
  - `・<行動>`
  - `→ <理由>`
- 曖昧表現禁止（例：「安静に」だけは禁止）。「何をどのくらい」が分かる具体動作＋軽い理由をセットで出す
- 理由行に `・` は使わない
- 番号付き（`1)` / `1️⃣`）は禁止
- 医療行為の指示は禁止。危険行為・専門処置は禁止

#### ③ 予想経過（安心設計）
- 多くの場合の一般的経過時間を示す（例：「数時間〜1日程度で徐々に落ち着いていくことが多い」）
- 断定禁止。不安を煽る表現禁止
- **口調**：本文全体と同様、**敬体（です・ます）**で統一する。

**経過時間による分岐（特例）**

- **通常**（経過が「さっき」「極短」「数分〜数十分」など、**ごく短い**と判断される場合）：上記のとおり、**数時間〜1日程度**のスケールで落ち着きの見通しを書く。
- **特例**（経過が**「1時間以上前」の類**に該当する場合。例：スロットや文脈が「数時間前」「半日以上」「一日以上前」「昨日から」「数日」など、**症状の始まりまたは話し始めから1時間を超えている**と解釈されるとき）：**「このまま安静にされていれば、数日かけて徐々に回復していくことが多いです。」**のように、**数日スケール**の見通しを述べる。**「安静」「無理をしない」**と**時間の目安（数日）**をセットで示し、安心感を優先する。
- 特例の文例（敬語・断定を避ける）：
  - 「このまま無理をせずお休みになられていれば、多くの場合は数日のうちに徐々に良くなっていきます。」
  - 「安静を続けられていれば、数日ほどで落ち着いてくることが多いです。」
  - 「急がず体を休めていらっしゃれば、だいたい数日かけて回復に向かうことが多いです。」
- 特例でも**断定禁止**（「必ず治ります」等は禁止）、**不安を煽る表現禁止**は通常と同じ。
- ユーザーの言葉（「3日前から」等）を勝手に変換しない。分岐は**スロット・箇条書きの経過情報**に基づく。

#### ④ 締めの一文（任意）
- 例：「今は体を回復モードに入れることが最優先です。」
- 心理的アンカーとして入れるとよい

### 10.2 表現ルール（厳守）
- **口調**：勧める口調を使う（`〜してください` / `〜するといいです`）。ユーザーにわかりやすく、具体的に伝える。
- 恐怖喚起禁止
- 医学用語の多用禁止
- 「今はこれで安全圏に入る」という安心設計を含意する
- ユーザーの言葉を勝手に変換しない
- ❌「安静にしてください」（抽象的）→ ⭕「体が回復モードに入れる環境を作ってください」
- ❌「水分を取りましょう」→ ⭕「脱水があると痛みが長引くため、少量ずつ補給するといいです」

### 10.3 生成パイプライン（必須）
- `🤝/📝「具体的に」` の生成内容とスロット情報を元に、症状文脈を抽出する。
- **今すぐやること**の検索クエリは、🤝今の状態についてのクエリ（主症状 + 箇条書き全文）に`対処法`を加えて作る（例：`頭痛 痛みは9/10 吐き気がある きっかけは太陽光と睡眠不足 対処法`）。
- **🔴の「受診までの過ごし方」も同じクエリ・パイプライン**を使用する（モーダル・🟢/🟡と同一）。
- **検索の強化**：
  - 日本語・英語それぞれに言語最適化クエリを複数用意し、痛み性状・部位を含むクエリを追加する。
  - 初回検索0件時は、追加クエリ（`主症状 対処法 セルフケア` / `主症状 自宅 ケア` / 英語対応）で再検索する。
  - 全クエリを並列実行し、結果を重複除去・ランク付けする。
- Google Custom Search を日本語・英語で実行する。429時はリトライ、num=10で取得。
- 検索結果は重複除去し、信頼ドメイン（公的機関・大学病院・大手医療情報サイト）を優先する。
- 上位結果の要点を仮説スコアに反映し、行動を決定する。
- 検索結果の生テキストをそのまま表示しない。
- **検索失敗時のフォールバック順**：1) 追加クエリで再検索 2) 検索を使わずLLMで文脈から生成 3) 汎用フォールバックの直前でLLMで再度生成を試行（ここで落ちることはない） 4) 汎用フォールバック（最悪の手段）
- **検索結果を要約して行動を生成する際も、勧める口調（〜してください／〜するといいです）を厳守する。**

### 10.4 仮説選択ルール（実装準拠）
- 特徴量（部位・痛み性状・経過時間・誘因・随伴症状・重症ヒント）で仮説を評価する。
- 主要一致は高配点、禁忌一致は減点する。
- 同点付近では安全側の仮説を優先する。
- OTC推奨は意味判断で決定し、禁忌がある場合は出さない。

### 10.5 理由の確実性（必須）
- 行動文は簡潔に書く（過度な詳細化は禁止）。
- 各理由は、Google検索で得た要点と整合する根拠を必ず含める。
- 理由は一般論だけで終わらせず、「なぜその行動が今の状態に合うか」を明示する。
- このルールは通常のまとめ本文（`✅ 今すぐやること`）と、`✅ 具体的に` モーダルの両方に適用する。

### 10.6 禁止出力（強制）
- 次の行動文は出力禁止：
  - `・症状メモを2時間ごとに1回、合計3回（強さ・変化・随伴症状）で記録し、同日中に悪化サインがないか再確認しましょう`
  - `→ 現在の状態データを再評価しやすくなり、次の判断の精度を維持できます。`

### 10.7 出力制約（必須）
- OTC（ワセリン等を含む市販品）は最大1件。
- OTCを無理に出さない（0件可）。**🟡のときは必ず1件OTCを勧める。**
- `水分を取る / 横になる / 安静にする` は必要に応じて迷わず出す。
- 診断の断定表現は禁止。
- 単なる検索結果の羅列は禁止。
- 参考情報を示す場合は、理由行内で `（参考: <ドメイン> / <ドメイン>）` の形式で示してよい。
- **理由文に検索の痕跡を残さない**：「検索結果より」「検索で」「検索情報では」「上位情報の」などの表現は禁止。理由を substantive に述べる（例：「刺激負荷を減らすことで、症状のぶれを抑えやすくなります」）。

### 10.8 フォールバック強化（必須）
- 検索・LLMが失敗した場合でも、**症状部位（頭・お腹・喉・皮膚）に応じた文脈補足**で埋める。
- **汎用フォールバックの前に必ずLLMで再度生成を試行する**。汎用フォールバックに落ちることはないようにする。
- 汎用フォールバック（「刺激を1つ減らし、水分を…」）は、LLMの再試行が失敗した場合の最終手段のみ。
- 部位別の補足候補を常に用意し、`ensureActionCount` で必ず3件以上を確保する。

## 11. 禁止事項（抜粋）
- 判断の丸投げ質問。
- 行動選択を促す質問。
- 質問フェーズでの助言・判断。
- まとめブロックの欠落。
- 表示上の箇条書きで「-」を使うこと（必ず「・」）。

## 12. 質問生成の内部要件
Kairoが質問を生成する際は、内部的に以下を必ず明確にする。
- この質問で埋める判断スロットは何か
- 直前のユーザー発言から1語以上拾えるか
- 「次に進むため」の目的宣言が入っているか

## 12.1 グローバル禁止（全フェーズ共通）
- 「ない／特にない／該当しない」を不安材料として扱う表現は禁止。
- 否定回答は常に「安心材料／リスク低下情報」として扱う。
- 例：「ないは気になります」「引っかかる」は絶対に使わない。

## 12.2 スロット再質問禁止（必須）
- 各スロットは 1 セッション 1 回のみ質問可能。
- 既に質問済みのスロットは再質問しない。
- 以降は「確認」ではなく「要約・反映」に切り替える。
- 質問導入文（「一点だけ伺います」「ここだけ確認します」等）は未質問スロットにのみ使用可。

## 12.3 まとめ生成ルール（必須）
- slotAnswersをそのまま列挙しない。
- **主症状・ユーザーの回答を「付随症状」として一括して扱わない**。それぞれを適切に区別する。
- 「・ない」「・特にない」単体の箇条書きを作らない。
- 必ず意味のある自然文に再構成する。
- 「日常生活では〜」等の前置きは使わない。取得内容をそのまま記載する。
- ユーザーが「〜です」と答えたものは、「です」を除いて自然な形にする。
- 箇条書き内で一度出たキーワードは繰り返さない。
- 判断・安心コメントは、直前の情報のうち1つ以上を根拠として反映する。

## 12.4 全フロー共通の回答解釈ルール（必須）
### 正規化オブジェクト
ユーザー回答は必ず以下に正規化する。生テキストをそのまま生成に使わない。
```
normalizedAnswer = {
  slotId: string,
  rawAnswer: string,
  riskLevel: "LOW" | "MEDIUM" | "HIGH"
}
```

### riskLevel 割当
- 各スロットの選択肢は事前に riskLevel を割り当てる。
- riskLevel 未定義の回答はエラー扱いとし、生成処理に進まない。

### テンプレート制御
- 文章生成は riskLevel 別テンプレートのみ使用可。
- LOW：安心材料／リスク低下のみ（不安・懸念語は禁止）
- MEDIUM：状況確認／軽い注意まで（断定・強い不安は禁止）
- HIGH：注意喚起／受診検討まで可（恐怖煽り・断定診断は禁止）

### 主観評価語の制限
- 「気になります／引っかかります／心配です／注意が必要です」は HIGH のみ使用可。
- LOW/MEDIUM では絶対に含めない。

### 矛盾防止チェック
- rawAnswer が「ない／特にない」の場合、不安・懸念語が含まれていないこと。
- riskLevel と文調が一致していること。
- 矛盾検出時は該当文を破棄し、同 riskLevel のテンプレで再生成する。

### 適用範囲
- 質問直後のリアクション文
- 状態まとめ（要約）
- 判断コメント
※ いずれも bypass 禁止

## 12.5 初回応答仕様（必須）
- ユーザーが最初に話したとき、安全文を質問の前に必ず表示する。
- 安全文は固定2パターンからランダムで1つ。質問とは分けて出力。安全文が先。
- 〇〇にはユーザーの主症状を短く簡潔にしたもの（頭がいたい→頭痛、唇が痛い→唇の痛みなど）。
- 初回の導入テンプレ（EMPATHY／PROGRESS／FOCUS）は**全て使用禁止**。質問のみ。

安全文テンプレ1:  
「慌てなくて大丈夫です。
多くの〇〇は命に関わるものではありません。
一緒に安全を固めていきましょう。」

安全文テンプレ2:  
「まず安心してください。
〇〇のほとんどは深刻なものではありません。
まず落ち着いて、ひとつずつ安全を確認していきましょう。」

## 12.6 共感テンプレの役割分離（必須）
- 共感テンプレは2種類のみ。
  1) EMPATHY_OPEN（初回のみ）
  2) EMPATHY_NEXT（2問目以降）
- フォールバック用の共感文は禁止。
- templateId / empathyTemplateId が無い場合、共感文は表示しない。
- 共感文はLLM生成禁止。
- questionCount により if/else で明示的に切り替える。
- EMPATHY_NEXT は「今まで通りの共感」を使う（ただし禁止語は除外）。


## 13. 判定設計の基本原則（必須）
- Kairoにおいて「判定完了」と「shouldJudge === true」は同義である。
- 判断スロットが6つ以上埋まった時点、または質問回数が8以上になった時点で、必ず judgeMeta.shouldJudge を true にする。
- **triage_state（必須）**：全APIレスポンスに `triage_state: { triage_level, is_final, required_fields_filled }` を含める。`is_final` が true になるまでサマリー・判定バナーを絶対に描画しない。
- フロントエンドは **triage_state.is_final === true** を唯一のトリガーとして、サマリーカード（🟢🟡🔴・確信度・比率・安心まとめ）を描画する。judgeMeta.shouldJudge は補助のみ。
- サマリーカードの文言は**15文字以内を厳守**。各色ごとに固定文を3つ用意し、ランダムで1つ表示。断定的すぎる医療判断は避け、安心感を優先した言葉にする。
  - 🟢：今は大きな心配なさそうです / 落ち着いて様子を見られそうです / 今のところ安心して過ごせそうです
  - 🟡：今は注意して様子を見てください / 少し注意しながら様子を見ましょう
  - 🔴：早めの受診をおすすめします / 医療機関での確認が必要そうです / 今日中の受診を検討してください
- サマリーカードとまとめブロックの緊急度表記（🟢🟡🔴）は必ず一致させる。矛盾はバグ扱いとする。
- **サマリーカードの色**：6スロット完了時にサーバー内部で算出した緊急度判定（triage_state.triage_level）を唯一の基準とする。クライアントはこれ以外の情報で上書きしない。
- shouldJudge === false の場合、サマリーカードは必ず非表示にする（表示禁止）。
- judgeMeta が存在するだけの状態でサマリーカードを表示してはならない。
- confidence や ratio は「説明用の指標」であり、UI描画の可否を分岐させるためのものではない。
- ⚠️ shouldJudge を false のまま判定完了させてはならない（UIが沈黙するため）。

## 14. 判断スロット設計（必須）
Kairoは以下6つの「判断スロット」を持つ。質問の有無に関係なく、ユーザー発話または質問応答で値が取得できた時点で「埋まった」と判定する。
1. 痛みの強さ
2. 悪化傾向
3. 経過時間
4. 日常生活への影響
5. 付随症状
6. 想定される原因カテゴリ（本人の認識）

### 14.0 固定順・固定文言（必須）
質問の形式は以下を維持する。未充填スロットのみ、次の固定順で1ターン1問ずつ出す。
（filled のスロットはスキップして次へ進む。6問必ず出す構造は禁止）
1. 痛みの強さ：  
　「今の痛みを、1から10であらわすと何点ですか？」
2. 悪化傾向（痛み方）：  
　「今の痛み方はどれに近いですか？」  
　・（AIが生成：症状に合った痛み方）  
　・（AIが生成：症状に合った痛み方）  
　・（AIが生成：症状に合った痛み方）
3. 経過時間：  
　「いつから始まりましたか」  
　・さっき  
　・数時間前  
　・一日前
3.5 悪化傾向（特例）：  
　「さっき」以外の回答（「数時間前」「半日」「昨日」「三日前」「一日以上前」など）が来た場合、3と4の間に挿入する。全カテゴリ共通。  
　「今の方向性はどうですか？」  
　・回復に向かっている  
　・変わらない  
　・発症時より悪化している  
　※「発症時より悪化している」選択時、かつ痛みの強さ（1問目）が5点以上の場合のみ必ず🔴とする。まとめにも出すこと。
4. 日常生活への影響：  
　「今の動ける感じはどれに近いですか？」  
　・普通に動ける  
　・少しつらいが動ける  
　・動けないほどつらい
5. 付随症状（PAIN・INFECTION の4問目）：  
　「これ以外の症状は他にありますか？」  
　**PAIN 未確定時**：・咳や鼻詰まりがある、・吐き気がある、・発熱がある  
　**元から INFECTION 系のとき**：・これ以外は特にない、・吐き気がある、・咳や鼻詰まりがある（「発熱がある」は出さない）
6. 想定される原因カテゴリ（本人の認識）：  
　「何かきっかけで思い当たることはありますか？」  
　・スマホやパソコンを長時間見た  
　・寝不足や疲れが続いている  
　・強いストレスや緊張があった

上記3択に当てはまらない自由記述は、従来の「自由記述回答の扱い（固定）」を適用する。
まとめでは「自由記述回答の扱い（固定）」を適用し、ユーザー原文を優先して記載する。

### 14.0 ユーザー回答の扱い（必須）
- **緊急度判定時以外**：全てのユーザー回答をそのまま、フィルタを通さずに使用する（まとめ・クエリ・会話履歴など）。
- **緊急度判定時のみ**：一番近い選択肢に合わせた数値化が必要。それ以外の用途では一切フィルタを通さない。
- **出力の型整形**：情報整理・まとめでは、ユーザーの自由記述をそのまま出さず、型に合わせて整形する（例：三日前→「三日前に症状が始まった」）。マッピングは緊急度判定の計算時のみ行う。
- **LLM出力の簡潔化**：「〜の可能性があります」→「〜の可能性」のように簡潔にするフィルタを適用する。

### 14.1 質問制御ルール
- 毎ターンの処理順序は以下で固定する：  
  `normalize_text() -> detect_symptom_type() -> extract_slot_values() -> update_slot_status() -> generate_next_action()`
- 毎ターン、ユーザー発話の解析→自動充填を先に実行する。
- 各質問は必ず、未充填スロットを1つ埋める目的を持つ。
- filled == true のスロットに質問することを禁止する。
- 同一スロットに2回質問することを禁止する。
- 1ターンで2問以上出すことを禁止する。
- 6スロットが全て埋まった時点で判定・まとめへ移行する。

### 14.1.1 正規化（必須）
- 全角数字は半角数字に正規化する。
- 「4点」「4くらい」「4/10」などは痛みスコア抽出時に `4` として扱う。
- 「きのう」は「昨日」に正規化する。
- 「かも」「と思う」等の曖昧語は削除しない（原文保持）。

### 14.1.2 抽出ルール（必須）
- severity：`1-10`、`◯点`、`◯/10` を抽出。
- worsening：`worsening/improving/stable/fluctuating` の trend と、痛み方 quality を分離して保持。
- duration：`raw_text` と `normalized` を保持し、表示は `raw_text` を使う。
- impact：生活機能への影響表現から抽出。
- associated：主症状以外を配列で保持。
- cause_category：原因推定語と不確実性表現（「かも」「と思う」）を保持。

### 14.1.3 上書きルール（必須）
- 既に filled のスロットは原則上書きしない。
- 明確な訂正意図（例：「やっぱ」「訂正」「言い直し」）がある場合のみ上書きを許可する。

### 14.2 slot_status（必須）
```
slot_status = {
  severity: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  worsening: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  duration: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  impact: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  associated: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  cause_category: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null }
}
```
- value にはユーザー原文を優先して保持する。
- 既に filled のスロットは上書きしない（明示的訂正時を除く）。

### 14.4 導入文テンプレ制御方式（必須）
導入文は **役割×バリエーション** のテンプレID制のみで出力する。  
LLMの自由生成は禁止。

#### 役割（role）
- EMPATHY（受け止め）
- PROGRESS（小さな前進）
- FOCUS（今見るポイント）

#### ルール
- 質問ブロックは「導入文（最大2文）→ 質問文 → 選択肢」の順。
- 各役割は3〜5個の文言バリエーションを持つ。
- 同一テンプレIDはセッション内で再利用禁止。

#### 質問スロット × 導入テンプレ対応表（必須）
1) 初回（痛みスコア）  
  - 安全文を質問の前に必ず出す（固定2パターンからランダム）。質問とは分けて出力。安全文が先。
    - 「慌てなくて大丈夫です。
    多くの〇〇は命に関わるものではありません。
    一緒に安全を固めていきましょう。」
    - 「まず安心してください。
    〇〇のほとんどは深刻なものではありません。
    まず落ち着いて、ひとつずつ安全を確認していきましょう。」
  - 〇〇にはユーザーの主症状を短く簡潔にしたもの（頭がいたい→頭痛、唇が痛い→唇の痛みなど）
  - 導入テンプレ（EMPATHY／PROGRESS／FOCUS）は**全て使用禁止**。質問のみ。  

2) 経過時間  
  - 使用可：FOCUS  
  - 禁止：EMPATHY／PROGRESS  

3) 悪化傾向  
  - 使用可：FOCUS  
  - 禁止：EMPATHY／PROGRESS  

4) 日常生活への影響  
  - 使用可：PROGRESS（セッション中最大1回）／FOCUS  
  - 禁止：EMPATHY  

5) 付随症状  
  - 使用可：FOCUS  
  - 禁止：EMPATHY／PROGRESS（既に使用済みなら不可）  

6) 原因・きっかけ  
  - 使用可：PROGRESS（未使用なら可）／FOCUS  

#### 最後の質問（必須）
- **最後の質問では何があっても**、PROGRESS／FOCUS を禁止。**質問文のみ**にする。

#### 再利用禁止ルール（必須）
- 共感／前進／目的の言い回しは直近2問と同じ表現を避ける。
- ①②③は直近5ターン以内の文・構文・語順・語尾・リズムの再利用を禁止。
- 同じ導入語・結びが続く場合は必ず言い換える。

#### 初回例外
- 初回は安全文＋質問のみ。導入テンプレ（EMPATHY／PROGRESS／FOCUS）は使用しない。
#### 追加ルール
- 1〜3問目は②小さな前進の言語化を必ず省略する。

### 14.5 質問導入文のテンプレート制御（必須）
導入文（①共感／②前進／③目的宣言）は、LLMの自由生成から完全に切り離す。
LLMは**テンプレIDのみ**を返し、文言はフロントで静的テンプレから挿入する。

#### LLM出力形式（固定）
```
{ "templateId": "FOLLOW_UP_SHORT", "question": "質問本文" }
```

#### 文タイプ（role）
- EMPATHY
- PROGRESS_CONFIRM
- NEXT_STEP_DECLARATION

#### テンプレート制御ルール
- 初回質問：安全文＋質問のみ（導入テンプレ不使用）
- 2問目以降：PROGRESS_CONFIRM は最大1回まで
- 同一テンプレIDは session 内で再利用禁止
- 選択済みテンプレIDは state で管理し、LLMには渡さない

#### 追加ルール
- 経過時間は数値で聞かない（◯時間・◯日は禁止）
- 「短期 or 持続」が分かれば十分
- ユーザーに思い出させたり考えさせる聞き方は禁止
- 共感文はユーザーの発言を自然な文に整形してから使う（「〜ですは」等の不自然文は禁止）

### 14.2 まとめとサマリー表示
- まとめ生成後も会話は継続する（FOLLOW_UP_STATE）。
- サマリーカードは、まとめブロックの先頭（🟢 ここまでの情報を整理します または 📝 今の状態について）を出すタイミングと**同時に**表示する。
- 判定後の表示順は必ず以下の順とする：
  1. サマリーカード
  2. 🟢🟡🔴の判定（まとめブロック先頭の見出しで表現）
  3. まとめブロック

### 14.3 質問生成ルール（追加）
- 判断スロットは固定だが、質問文は固定してはいけない。
- 質問文は直前のユーザー発言・感情・文脈を必ず反映して生成する。
- 同じ判断スロットでも、会話の流れによって質問の聞き方を変える。
- テンプレ質問を連続して使用することは禁止。
