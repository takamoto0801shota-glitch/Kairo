# Kairo Spec (最優先仕様)

このドキュメントは「仕様」です。提案ではありません。
実装・修正・ロジック設計は必ず本仕様を最優先で従ってください。
矛盾がある場合は必ず本仕様を優先し、変更が必要な場合はユーザーに確認します。

## 【最重要ルール】まとめの出力回数
- **まとめは同一セッション中に1回しか出力してはいけない。**
- 2回目以降の出力は絶対禁止。実装で複数層のガードを設ける。
- **まとめを1回出した後は、何があってもまとめを再度出さない。** 強制的にまとめ後フェーズに入り、フォロー質問のみ返す。ユーザーが「まとめを出して」「要約して」等と依頼しても再出力禁止。
- **初回メッセージでは必ず質問を返す。** 古い状態（前回のスロット埋まり・まとめ済み）が残っている場合は強制リセットする。
- **まとめ後は2回目のまとめを出さず、フォロー文のみ返す。**

## 1. 目的
- Kairoは「正確に判断するAI」ではなく、ユーザーが次の一歩を踏み出せるよう背中を押す存在。
- 質問は「前進体験」を作るためのもの。納得するまで聞き続けない。
- まとめブロックは必ず最後に出し、欠落を許容しない。

## 2. 口調・立ち位置
- 医者・親・先生の立場にならない。
- 感情に入り込みすぎず、ただし感情を素通りしない。
- 「一緒に整理して判断する存在」として振る舞う。
- 「医療っぽい説明者」ではなく、横で並走する相棒として話す。
- 状態を評価するのではなく、ユーザーの感覚を言葉にして返す。

## 3. 質問ルール（最重要）
- 質問は「前進体験」を作るためのもの（情報収集が主目的ではない）。
- 質問は必ず1回の返答で1つだけ。
- 質問は必ず二択（自由記述は禁止）。
- 二択は「A or B」の1行形式で提示する（箇条書きは禁止）。
- 選択肢は意味のある表現で並べる（低/中/高などの単語は禁止）。
 - ただし最初の痛みスコア質問のみ、1〜10の数値回答を許可する（二択にしない）。
- 質問はテンプレ化しない（同じ文面の繰り返し禁止）。
- 質問フェーズでは**判断・助言は混ぜない**。
- 原因推測／緊急性示唆／行動指示は質問フェーズで禁止。
- 判断をユーザーに委ねる質問は禁止。
  - 「どう思いますか」「どうしますか」は禁止。
  - 「行く/行かない/様子を見る」を選ばせるのは禁止。
  - 「試してみたいことはありますか」など行動選択質問は禁止。

## 3.1 禁止質問パターン
以下の質問・選択肢は使用禁止とする。

- 痛みの強さを主観で聞く質問
  - 例：
    - 軽い／中くらい／強い
    - どのくらい痛いですか？

理由：
- 不安による誇張が起きる
- スコアに正規化できない
- 判定精度を下げる

## 3.2 痛み評価の質問形式（必須）
痛みの評価は必ず「行動可能性・日常への影響」で聞く。
質問文はテンプレ化しないが、内容は行動可能性ベースにする。

## 4. 質問回数の下限・上限
- 最低質問回数は5回。
- 質問の上限は設けない（6スロット充填を最優先）。
- まとめに移行する条件は「判断スロットが6つ全て埋まった時点」のみ。
- 6スロットが未充填のまま判定＋まとめへ進むことを禁止する。

## 5. 最後の質問
- Kairoが「最後の質問」と判断した場合、質問文頭は
  「最後に」または「最後の質問です」で開始する。

## 6. 確信度（confidence）の定義
- confidence は「結果」として算出する（判定条件ではない）。
- 算出式：
  - confidence =（埋まった判断スロット数 / 6）× 100
- confidence は UI表現や文言の強さ調整にのみ使用する。
- confidence の数値で判定・まとめ・サマリー表示を制御してはいけない。

## 7. 緊急度判定（危険フラグ優先モデル）

緊急度判定は2段階構造とする。

### Phase1：即時RED条件
以下のいずれかを満たす場合、比率計算を行わず🔴とする。
1. pain_score が高（7以上）かつ daily_impact が高
2. pain_score が高（7以上）かつ associated_symptoms が中以上
3. daily_impact が高 かつ associated_symptoms が中以上
4. criticalスロット（pain_score / daily_impact / associated_symptoms）のうち、高レベル（最大weight=3）が2つ以上
5. criticalスロット（pain_score / daily_impact / associated_symptoms）のうち、高レベルが1つだけの場合は必ず🟡（2つ以上は従来どおり）

### Phase2：重症指数計算
各項目を以下でスコア化：
- 低 = 0
- 中 = 1
- 高 = 3

スロット別のレベル定義（固定）：
- pain_score（数値）
  - 1〜4 = 低
  - 5〜6 = 中
  - 7〜10 = 高
- 日常動作影響
  - 「普通に動ける」= 低
  - 「少しつらいが動ける」= 中
  - 「動けないほどつらい」= 高
- 追加症状
  - 「なし」= 低
  - 「吐き気・めまい」= 中
  - 「しびれ・視界異常」= 高
- その他スロット（痛みの質 / 発症タイミング / 原因カテゴリ）は、質問の選択肢順（1つ目=低、2つ目=中、3つ目=高）で固定する。

自由記述回答の扱い（固定）：
- **緊急度判定時のみ**：選択肢に完全一致しない回答でも、意味が最も近い選択肢のスコアを適用する（マッピング）。それ以外の用途ではマッピングしない。
- **出力は型に合わせる**：自由記述をそのまま出さず、あらかじめの型に合わせて整形する。例：「三日前」→「三日前に症状が始まった」、「少しつらいが動ける」→「日常生活は少しつらいが動ける程度である」。

重要度係数：
- pain_score ×1.4
- daily_impact ×1.0
- associated_symptoms ×1.0
- onset ×1.0
- quality ×1.0
- cause ×0.8

重症指数：

severityIndex =
（各スコア × 係数 の合計）÷ 18.6

判定基準：
- 0.65以上 → 🔴
- 0.4〜0.64 → 🟡
- 0.4未満 → 🟢

### RED抑制ガード（PAINカテゴリ限定・必須）
- `PAIN`（痛み系）で、経過時間（スロット3）が「さっき」または「数時間前」の場合、**🔴判定を禁止**する。
- このガードは Phase1 / Phase2 の両方に適用する（最終判定を🔴にしてはいけない）。
- 上記条件に該当する場合、最終判定は `🟡` または `🟢` のみ許可する。
- カテゴリ判定は会話開始時に確定したカテゴリ（`triageCategory`）を優先し、後続の文脈混入でPAIN判定が揺れても、このガードを維持する。

指数や内部計算はユーザーに表示しない。

## 7.1 6スロット固定（動的充填）
判定に使うスロットは以下の6つで固定する（外部仕様は変更しない）。
1. pain_score（1〜10）
2. 痛みの質
3. 発症タイミング
4. 日常動作影響
5. 追加症状
6. 原因カテゴリ

### 7.1.1 症状カテゴリによる質問差し替え（固定）
症状は以下の4カテゴリに分類する：
- PAIN（痛み系）
- SKIN（皮膚粘膜系）
- INFECTION（発熱感染系）
- GI（消化器系）

カテゴリ判定キーワード（例）：
- PAIN: 頭痛 / 傷
- SKIN: ヒリヒリ / かゆい / 赤い / 発疹 / 唇 / 水ぶくれ / 乾燥 / 口
- INFECTION: 熱 / 発熱 / だるい / 咳 / 喉 / 寒気 / 風邪
- GI: 腹痛 / お腹 / 下痢 / 吐き気 / 胃 / 嘔吐 / 便

質問差し替えルール：
- PAINは従来の質問を維持する。
- SKIN / INFECTION / GI の場合は、スロット4・5・6の質問文と選択肢をカテゴリ定義に差し替える。
- 各質問は3択で提示し、選択肢は低リスク→中リスク→高リスクの順に並べる。
- 自由記述は「最も近い選択肢」にマッピングしてスコア計算に使用する（内部計算は従来どおり）。

#### カテゴリ別 質問テンプレ（固定）

**PAIN・INFECTION 特例（最初は区別できない）：**
- 4問目（スロット4）を付随症状にする。PAIN・INFECTION 共通。
- 質問：「これ以外の症状は他にありますか？」
- 選択肢（固定）：・これ以外は特にない、・吐き気がある、・咳や発熱がある
- **喉が痛いなど、喉的なものが主症状の時**：必ず初めからずっと INFECTION 系とする（4問目を待たずに最初から INFECTION）。
- **頭が熱い等をユーザーが先に言っていた場合**：・咳や発熱がある → ・咳がある のみに変更
- **咳と発熱のどちらか、またはどちらも選んだ場合**：必ず INFECTION 系カテゴリとする。
- 4問目で咳または発熱的な回答（咳、咳や発熱、頭が熱い等）をした場合 INFECTION とする。
- 他の自由記述があっても、咳または発熱的な回答の時は INFECTION。

**PAIN（痛み系）**：スロット5（日常生活への影響）を削除。スロット6を最後の質問とする。
- スロット4: 「これ以外の症状は他にありますか？」（上記の選択肢）
- スロット6: 「何かきっかけで思い当たることはありますか？」（最後の質問。既存ルールに従い「最後に〜」から始める）
  - スマホやパソコンを長時間見た
  - 寝不足や疲れが続いている
  - 強いストレスや緊張があった
  - ※PAINカテゴリ中はこの3択を固定し、他カテゴリ（GI等）のスロット6選択肢へ差し替えない（強制）。

SKIN（皮膚粘膜系）：
- スロット数は5個（スロット6は削除）。
- スロット4（見た目）: 「見た目の変化はありますか？」
  - 赤みや乾燥だけ
  - 見た目はほとんど変わらない
  - 水ぶくれ・ただれ・できもの
- スロット5（きっかけ）: 「思い当たるきっかけはありますか？」（最後の質問。「最後に〜」から始める）
  - 特に思い当たらない
  - 紫外線や乾燥が強かった
  - 新しい製品や刺激物を使った

INFECTION（発熱感染系）：喉が痛い等の喉主症状の場合は初めから INFECTION。それ以外は4問目で咳・発熱的回答時のみこのカテゴリになる
- スロット4（付随症状）: 「これ以外の症状は他にありますか？」（PAIN と同じ選択肢）
- スロット5（体温）: 「体温はどのくらいですか？」
  - 平熱に近い
  - 37度台
  - 38度以上
- スロット6（きっかけ）: 「何かきっかけで思い当たることはありますか？」
  - 思い当たらない
  - 周りが咳をしていた
  - ストレスや疲労
- ※主症状（喉の違和感や鼻水等）の質問は削除

GI（消化器系）：
- スロット数は5個（スロット4は削除）。
- スロット5（便や吐き気）: 「便や吐き気はどうですか？」
  - 特に変化はない
  - 下痢がある
  - 吐き気・嘔吐がある
- スロット6（きっかけ）: 「何かきっかけで思い当たることはありますか？」
  - 冷えや寒気がある
  - 便秘
  - 食あたり

カテゴリ運用ルール（強制）：
- カテゴリは会話開始時に確定した `triageCategory` を固定して使う。
- 質問途中で文脈が混ざっても、同一セッション中は質問カテゴリを切り替えない。

### 発話優先原則（必須）
- ユーザー発話にスロット情報が含まれる場合は、そのターンで自動充填する。
- 自動充填されたスロットは対応質問をスキップする。
- 既に埋まったスロットは再質問しない。
- 未充填スロットは優先順で必ず確認する（飛ばさない）。

### 内部管理（必須）
```
slot_status = {
  severity: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  worsening: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  duration: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  impact: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  associated: { filled: boolean, source: "user_spontaneous" | "question_response" | null },
  cause_category: { filled: boolean, source: "user_spontaneous" | "question_response" | null }
}
```

### 質問生成ルール（必須）
- 1ターンで質問は最大1つ。
- 対象は未充填スロットのみ。
- 優先順：pain_score → worsening → duration → daily_impact → associated_symptoms → cause_category。
- 充填済みスロットの再質問は禁止。
- 6スロットが全て埋まるまで、未充填スロットへの質問を継続する（強制）。
- 6スロット未充填でまとめへ遷移することを禁止する。

## 7.2 Rest & MC Decision Layer（後段レイヤー）
目的：
- 既存の緊急度判定（Phase1/Phase2）には一切影響させない。
- 医学的判断と社会的対応（MC）を分離して提示する。
- 6問固定仕様は変更しない。

### 6スロット固定（質問は動的）
以下の6スロット自体は固定。質問は「未充填スロットのみ」に対して出す。
1. pain_score（1〜10）
2. 痛みの質
3. 発症タイミング
4. 日常生活への影響
5. 追加症状
6. 原因カテゴリ

### 4問目：日常生活への影響（休息判定専用スロット）
質問文（固定）：
- 「今の動ける感じはどれに近いですか？」

選択肢（固定・順番固定）：
1. 普通に動ける
2. 少しつらいが動ける
3. 動けないほどつらい

### 休息判定（daily_impact のみで決定）
- 「普通に動ける」→ restLevel = NONE
- 「少しつらいが動ける」→ restLevel = LIGHT
- 「動けないほどつらい」→ restLevel = STRONG
- 他スロットは休息判定に使わない。
- 痛みスコア（1問目）を休息判定に使わない。
- 付随症状（5問目）を休息判定に使わない。
- restLevelは後段の Rest & MC Decision Layer 専用変数とする。

### カテゴリ増加後の休息判定ルール（固定）
- `PAIN`：スロット5（日常生活への影響）を削除したため、`restLevel = LIGHT` で固定（🔴のときは休息・MCブロックを出さない）
- `SKIN`：常に `restLevel = NONE`
- `INFECTION`：`@kairo/KAIRO_SPEC.md:184-187`（体温3択）を 1:1 で休息判定に使う
  - 「平熱に近い」→ `NONE`
  - 「37度台」→ `LIGHT`
  - 「38度以上」→ `STRONG`
  - 仕組みは `@kairo/KAIRO_SPEC.md:261-268` と同じ（他スロットは休息判定に使わない）
- `GI`：常に `restLevel = LIGHT`

### MC必要性判定
- restLevel = NONE → mcRecommended = false
- restLevel = LIGHT → mcRecommended = optional
- restLevel = STRONG → mcRecommended = true

### 緊急度 × 休息 の最終行動分岐
- urgency = 🟢
  - NONE：自宅様子見
  - LIGHT：自宅休息（会社規定で必要ならオンライン診療可）
  - STRONG：自宅休息（MCが必要ならオンライン診療推奨）
- urgency = 🟡
  - NONE：市販薬推奨
  - LIGHT：市販薬＋休息（MCが必要ならオンライン診療で取得可能レベル）
  - STRONG：市販薬＋強い休息（MC取得目的ならオンライン診療を第一選択）
  - ※ 🟡では、休息理由だけで対面受診推奨に切り替えない。
- urgency = 🔴
  - 行動：対面受診を優先（受診優先度は変更しない）
  - MC取得は副次目的として扱う
  - 休息・MCブロック（🧾）は表示しない

### 出力原則
- 「医学的判断」と「社会的対応（MC）」を分離して説明する。
- オンライン診療は社会的合理性の選択肢として提示する。
- 医学的緊急度は上書きしない。

### 役割分離（固定）
- 緊急度判定：全6問
- 休息判定：4問目のみ

### 意味定義（固定）
- NONE：勤務可能レベル
- LIGHT：勤務は可能だが休息が望ましい
- STRONG：勤務は困難と判断

※ restLevel は医学的重症度ではなく、機能的制限レベルを表す。

## 8. まとめブロック（必須）
- どんな場合でもまとめブロックは必ず出す。
- まとめブロックの欠落は絶対に許容しない。
- **【最重要】まとめブロックは同一セッション中に一度だけ出力する。2回目以降の出力は絶対禁止。**

### 8.0 まとめ前の確認（必須・🟢🟡🔴共通）
- 6スロット全て埋まったら、**いきなりまとめを出さず**、まずユーザーに確認を取る。
- 確認文はまとめブロック形式ではなく、**普通の質問の形**で出す。
- **表示アニメーション**：確認メッセージはすぐに出さず、上から下へスライドインするアニメーションで表示する。
- 構成：
  1. 導入：「今のところ整理できているのは、」
  2. 箇条書き：6スロットの取得内容をそのまま使う（最大8行）。「日常生活では〜」等の前置きは使わない。`@kairo/KAIRO_SPEC.md:343` のルールに従う。必ずユーザーの情報を落とさないこと。
  3. 共感文＋接続語＋判断文：**生成禁止**。固定テンプレのみ。接続語をつけて、既存のテンプレ文をランダムで入れるだけ。
    - **共感テンプレ**（経過時間で選択）：固定テンプレからランダムで1つ。
      - 「さっき」に近い選択肢の場合：① 急に症状が出ると、「何か大きなことかも」と不安になりますよね／② 突然の変化だと、悪い方向を想像してしまいますよね
      - 「数時間前」「一日前」に近い選択肢の場合：① なかなか良くならないと、「何か隠れているのでは」と心配になりますよね／② なかなか良くならないと、何か深刻なのではと心配になりますよね／③ ずっと続いていると、だんだん不安が大きくなりますよね
    - **接続語**：判断文の直前に緊急度で挿入。🟢は「ただ、」／🟡は入れない／🔴は「なので、」
    - **判断文**：固定テンプレからランダムで1つ。文章の追加・生成は禁止。
      - 🟢：① 今の症状の出方からは、ひとまず大きな心配をしなくてもよさそうです／② 今お聞きしている範囲では、過度に不安になる必要はなさそうに見えます／③ 今の状態からは、急いで何かをしなければならない状況ではなさそうです
      - 🟡：① 大きく慌てる状況ではなさそうですが、念のため体調の変化には気をつけておきたい状態です／② 今のところ深刻なサインは見えていませんが、無理をせず経過を見ていくことが大切そうです
      - 🔴：① 念のため早めに医療機関で確認しておいた方がよさそうです。落ち着いて行動すれば大丈夫です／② 安心のためにも今日中の受診を考えておきたい状態です／③ 一度専門家に確認してもらう方が安心につながりそうです
  4. 確認文：必ず「合っていますか？」または「これでよろしいですか？」など確認系の文を入れる。その後に「もしまだ足りないことがあれば教えてください」的な文を必ず入れる。固定ではなく毎回生成し、バリエーションを持たせる。
- **付随症状が最初の発言で言われた場合**：確認文・まとめには記載しない。他のスロットが補充された場合は今の形に合うようにLLMで要約する。
- ユーザー応答の扱い：
  - **確認文でユーザーが何かしらの返事をした場合は必ずまとめへ移行する**。
  - **追加情報**（自由記述で症状・経過などを補足した場合）→ まとめブロックの「今の状態について」のみ差し替え（他ブロックはそのまま）。その情報を箇条書きに追加。クエリ検索時も使う。
  - **それ以外**（「あっている」「大丈夫です」「はい」等の肯定のみ）→ **絶対に箇条書きには書かず**、まとめを出す。
  - **違う／間違っている**→ まとめを必ず出す（中間メッセージは出さない）。
- **確認文の後にユーザーが何かしら言ったら、まとめをいつでも出してよい。必ずまとめを出す。それ以外のもの（中間メッセージ等）は強制的に出さない。**
- **ユーザーの言葉の仕分け**：肯定のみ（あっている・大丈夫・ない・特にない・はい等）→ 既存まとめを使用。追加情報（症状・経過等を補足）や違う等→ 「今の状態について」ブロックのみ差し替え（他ブロックはそのまま）。
- この順番を必ず守る：確認 → ユーザー応答 → まとめ。
- **確認文は出すだけ。まとめは「全スロット埋まり」または「質問が全て完了（7問）」のどちらかが満たされた時点で、強制的に生成を開始する**（バックグラウンド先行生成は行わない）。
- **全スロット完了時、または質問完了時**：確認文表示と同時にまとめ生成を開始。ユーザー応答時は生成完了を待ってから表示。
- ユーザーが応答したら、**まとめを出すだけ**。中間メッセージ・閉じメッセージ・LLM応答等は一切出さない。
- まとめの出力は**ユーザーが確認文に対して何かを言った後**のみ行う。それまでは待機。
- ユーザー応答時、まだまとめ生成が終わっていなければ**完了まで待ってから出力**する。
- **サマリーカードと共にまとめブロックを順に生成**する（既存のルールのまま）。
- **フォロー文はまとめを全て出し終えてから出力する**（どんな場合でも）。
- **まとめの後にはフォロー文を絶対表示する**。
- まとめブロック内に**質問文を一切書かない**（疑問文・?・？は禁止）。
- まとめブロックの後は**フォロー質問フェーズ**に必ず移行する（会話終了にしない）。
- まとめ内容はテンプレ禁止。会話内容に即して毎回生成。

#### 🤝 今の状態について（🟢/🟡/🔴・確認文共通）

**適用範囲**：🟢/🟡のまとめブロック、🔴の📝今の状態について、まとめ前の確認文。いずれも同じ設計で生成する。

**確認文とまとめの出力形式の違い**：
- **確認文**：「今の情報から見ると、」+ 箇条書き +「という状況です。」の形
- **まとめの「今の状態について」**：箇条書きのみ。「今の情報から見ると、」と「という状況です。」は出さない

**目的**：転記ではなく**解釈**で生成する。ユーザーの情報を医療的に意味のある形に整理し、「なるほど、こういう状態なんだ」と理解→安心につなげる。

**LLM理解レイヤー（重要）**

情報整理は単なる言い換えではなく、**症状の状態を説明する文章**として生成する。

| 禁止 | OK |
|------|-----|
| 回答をそのまま並べること | 症状の意味を短い文章で説明する |
| ・ズキズキする | ・ズキズキするタイプの痛みが出ている |
| ・さっき始まった | ・症状は急に始まっている |

**処理順序**：回答 → 状況理解 → 要約

**生成プロセス**（LLMは必ずこの思考順序で生成）：
1. **Step1** ユーザーの回答を整理する
2. **Step2** それぞれの回答が示す「症状の意味」を考える
3. **Step3** その意味を短い文章にする

**例**：入力（痛み6/10、ズキズキ、さっき、吐き気なし）→ 思考（痛みはやや強い、鋭い痛み、急性発症、他症状なし）→ 出力（・痛みはやや強め（6/10）、・鋭いタイプの痛みが出ている、・症状は急に始まっている、・吐き気や発熱などの他の症状は今のところ見られていない）

**出力フォーマット（固定）**：
- **確認文**：「今の情報から見ると、」+ 箇条書き +「という状況です。」
- **まとめ**：箇条書きのみ（「今の情報から見ると、」「という状況です。」は出さない）

```
確認文：
今の情報から見ると、

・〇〇
・〇〇
・〇〇
（最大8項目）

という状況です。

まとめの「今の状態について」：
・〇〇
・〇〇
・〇〇
（最大8項目）
```

**禁止事項**：ユーザーの文章をそのまま表示。「特にないです」などの会話文。断定表現（「可能性」「今のところ」を使う）。

**各スロットの変換ルール**（入力→症状の状態を説明する文章）：

| スロット | 入力例 | 出力例（症状の意味を説明） |
|---------|--------|---------------------------|
| 痛みの強さ | 1〜3 | ・痛みは軽め（〇/10） |
| | 4〜6 | ・痛みは中程度（〇/10） |
| | 7〜8 | ・痛みはやや強め（〇/10） |
| | 9〜10 | ・痛みは強め（〇/10） |
| 痛み方 | ズキズキする | ・ズキズキするタイプの痛みが出ている |

| | 一日前 | ・症状は一日ほど前から続いている |
| 悪化傾向※ | 回復に向かっている | ・症状は回復方向に向かっている |

**箇条書きフィルタ**：誤りや日本語として不自然な箇条を検出し、修正する。修正可能な場合はそこだけ直し、修正不可の場合は除外する。重複助詞（はは、がが等）、重複接尾（ようなような等）、不完全な文末（助詞で終わる等）を検出・修正する。

**重要ルール**：断定しない（「可能性」「今のところ」を使う）。ユーザーの言葉をそのまま表示しない。1項目1行。最大8項目。

- **絶対ルール**: ユーザーの文を第一優先とする。選択肢の全文でユーザー発話を置き換えない。ユーザーが言ったことのみを使う。
- 「ない / わからない / 不明 / 思い当たらない」等の回答は表示しない。
- 複数要素（例：太陽光と睡眠不足）が1回答に含まれる場合は、まとめ行でも必ず優先して保持する。
- `source`（`user_spontaneous` / `question_response`）は内部管理のみ。ユーザー表示は禁止。
- 箇条書きの後に改行を挟む
- 2文のみ（3文以上は禁止）
- 構成：事実 → 判断（感想・解説は禁止）
- 診断・断定・医療判断は禁止
- 「だから」「なので」等の接続語で判断を明示
- 「〜かもしれません」は最大1回まで
- 行動指示禁止

#### 🤝 今の状態について（🟢/🟡・判断文の型）
- **目的**：ユーザーに安心感を与えるため、判断文は必ず次の構造にする。
  1. 危険サインがないこと
  2. ユーザーへの共感
  3. 症状情報を1つ理由として入れる（`{symptomInfo}`）
  4. 様子を見る判断
- この構造を固定したテンプレートを用意し、🟢用と🟡用で分けて使用する。
- `{symptomInfo}` にはユーザーの症状情報を1つ入れる。以下から**緊急度が低いものから1つ**選ぶ（嘘のフォールバック禁止）：
  - 回復に向かっている
  - 他に強い症状が見られていない
  - きっかけは〇〇の可能性がある（**🟢の場合のみ**症状情報として使用可）
  - 症状が先ほどから始まった（ユーザーが「さっき」を選んだ場合のみ）
- 上記4つに当てはまらない場合：`{symptomInfo} ことから、` または `{symptomInfo} ため、` を削除して表示する（該当部分を省略）。
- symptomInfo の例：「回復に向かっている」「他に強い症状が見られていない」「きっかけはスマホの長時間使用の可能性がある」（きっかけは🟢のみ）「症状が先ほどから始まった」（さっき選択時）
- 実装ルール：該当する症状情報があれば `{symptomInfo}` に1つ入れる。該当なしの場合は `{symptomInfo} ことから、` / `{symptomInfo} ため、` を削除して表示。**「。」の直後に改行を1つ入れる。空行は作らない**。AIの説明ではなく、ユーザーに語りかける自然な文章にする。

- **🟢専用テンプレート**（4つ、ランダムで使用）：
  - ① 今の情報を見る限り、急いで受診する必要がありそうなサインは見当たりません。
    痛みはつらいと思いますが、{symptomInfo} ことから、今は大きく心配する状況ではなさそうです。
  - ② 現在の症状から見ると、緊急性が高そうなサインは今のところ見られていません。
    不安になると思いますが、{symptomInfo} ため、今の状態は大きな心配はなさそうです。
  - ③ 現時点の情報では、危険なサインは特に見当たりません。
    つらい症状があると不安になると思いますが、{symptomInfo} ことから、今は大きく心配する状況ではなさそうです。
  - ④ 今の症状を総合して見ると、急いで受診が必要な状況ではなさそうです。
    痛みや違和感があると心配になりますよね。
    ただ、{symptomInfo} ため、今の状態は大きな心配はなさそうです。

- **🟡専用テンプレート**（3つ、条件に応じて選択）：
  - ① 今の情報を見る限り、危険なサインは今のところ見当たりません。
    つらい症状があると不安になりますよね。
    ただ、{symptomInfo} ことから、今すぐ大きく心配する状況ではなさそうです。
    体調の変化には少し気をつけながら様子を見ていきましょう。
    → **痛みが7以上の時だけ**使用可能
  - ② 現在の症状から見ると、緊急性が高そうなサインは今のところ見られていません。
    体調がいつもと違うと心配になりますよね。
    ただ、{symptomInfo} ため、今の状態は深刻な状況ではなさそうです。
    無理をせず、体調の変化を見ながら過ごしてみてください。
    → いつでも使えるが、**優先度が一番低い**（フォールバック）
  - ③ 現時点の情報では、危険なサインは特に見当たりません。
    症状が続くと不安になりますよね。
    ただ、{symptomInfo} ことから、今すぐ大きな問題につながる可能性は低そうです。
    体調を気にかけながら様子を見ていきましょう。
    → **経過時間が「数時間前・一日前」を選択した時のみ**使用

##### 🤝/📝 LLM安全フィルタ仕様
- 出力フォーマット固定：`{ "common": [], "conditional": [], "rare_emergency": [] }`
- **common**：最低2件・最大4件。**conditional**：最低2件・最大4件。**rare_emergency**：**2件のみ**（強制）。
- **common** = 一般的に頻度が高い原因。`・<原因名> → <理由>` の理由は**ユーザーの言動を要約**して記載。固定文にしない。
- **conditional** = 条件付きで考慮。`・<病名> → <関連した理由>` 形式で記載。**本当の病名・疾患名**を使い、理由はユーザー症状と関連付ける（例：群発頭痛 → ズキズキする痛みが目の奥に集中することがある）。検索結果＋症状まとめから要約。固定テンプレート禁止。
- **キーワード重複禁止**：一度出たキーワード（ズキズキ・痛み・頭痛 等）は、同じ箇条書き内で繰り返さない。
- **rare_emergency** = 稀だが緊急性あり（デフォルト非表示）。**2件のみ**（強制）。検索結果＋症状まとめから要約。固定テンプレート禁止。
- 「今の状態について」の箇条書きを参照し、ユーザーが言っていないことは記載しない。

##### 🤝/📝 禁止事項
- **理由に痛みの強さ（3/10、〇/10など）を絶対に使わない**。箇条書きの痛みスコアを理由に含めない。

##### 🤝/📝 「→」理由のチェック・置換（必須）
- **チェック**：理由文中に「ユーザーが言ったと仮定される語」（肩こり・ストレス・寝不足・脱水・光・音・疲労 等）が含まれるか検出する。
- **置換**：ユーザーが言っていない語があれば、**ピンポイントで**ユーザーが言った言葉に置き換える。箇条書き全体を要約するのではなく、「→」理由内の該当語のみを置換する。

##### 🤝/📝 UI表示ルール（強制）
- **🟢 common**：常時表示。安心文を必ず添える（例：「頭痛の多くは生活要因や疲労によるものです」）。
- **🟡 conditional**：常時表示。煽らない表現にする。
- **🔴 rare_emergency**：初期状態では折りたたみ。表示条件：`triage_level == red` または ユーザーが「強い症状がある」を選択。
- **危険ワード初期表示禁止**：腫瘍・出血・致死・がん・破裂。これらは rare_emergency セクション内のみ表示可。

##### 🤝/📝 モーダルの表示仕様
- ユーザーの症状・入力内容は文章に含めない。
- 「あなたの場合」「この症状は」などの個別化表現は禁止。
- 診断を断定する表現は禁止。
- 恐怖を煽る表現は禁止。

##### ✅ モーダル（別仕様）

  - 休むためにMCが必要な場合は、今の症状であればオンライン診療で容易に取得できます。

  4. 各施設：理由を簡単に2つ、営業中かどうか、星評価を表示する

##### 🤝/📝 モーダルの表示構造（固定）
- `あなたの状態の理解を深める`（見出し固定）
- **🟢 よくある原因**：2〜4件。`・<原因名> → <理由>` 形式。理由は**ユーザーの言動を要約**して記載（固定にしない）。
- **🟡 状況によっては確認が必要**：2〜4件。`・<病名> → <関連した理由>` 形式。本当の病名を使い、理由はユーザー症状と関連付ける。検索＋症状まとめから要約。煽らない表現。
- **🔴 すぐ受診が必要なサイン**：**2件のみ**（強制）。折りたたみ・初期非表示。「強い症状がある場合はこちら」で展開。
- **🟢/🟡のとき**：`現時点の安心材料` → `こんな変化があれば受診を検討`（最大3）
- **🔴のとき**：`今回受診をおすすめしている理由`（見出し固定）
  - 箇条書き3件：ユーザーの状態から抜き出した事実を要約して記載（例：症状が4日前から続いている、痛みが10/10と強い、発症時より悪化している）
  - 締めの一文（固定）：「これらがあるため、一度医療機関で確認しておくと安心です。」

##### 🤝/📝 出力例（ベスト）
- **🟢**：2〜4件。**🟡**：2〜4件、`・<病名> → <関連した理由>` 形式。**🔴**：**2件のみ**（強制）。
- **禁止**：理由に痛みの強さ（3/10など）を絶対に使わない。
```

・偏頭痛 → ズキズキする痛みが特徴で、目の奥に痛みを感じることがある。
・緊張型頭痛 → 日常生活に影響を与えるが、動ける程度の痛みがある。
・熱中症 → 太陽をかなり浴びたことがきっかけで、だるさを感じている。

🟡 状況によっては確認が必要
・群発頭痛 → ズキズキする痛みが目の奥に集中することがある。
・副鼻腔炎 → 目の奥の痛みとだるさが関連している可能性がある。

頭痛のほとんどは命に関わるものではありません。特に、急激な悪化や神経症状がなければ、よくあるタイプの可能性が高いです。

現時点の安心材料
・経過が比較的短時間で、変化を追いやすい

こんな変化があれば受診を検討
・痛みが急に強くなる
・しびれや視界の違和感が新たに出る
・日常動作が急に難しくなる
```

（🔴 すぐ受診が必要なサイン：2件のみ・折りたたみ・初期非表示 → くも膜下出血、脳出血 等）
- **重要**：ユーザーが肩こり・ストレスを言っていない場合は「肩こりやストレスで」と書かない。ユーザーの言動を要約して理由を記載する。
- 🔴時：「今のあなたは🟡の可能性もないとは言えません。なので、確認をするためにも受診をおすすめします。」
- 🟢/🟡時：「〇〇（主症状）のほとんどは命に関わるものではありません。特に、急激な悪化や神経症状がなければ、よくあるタイプの頭痛の可能性が高いです。」
  改行のうえ、**この症状はよくあるものと思わせる文を必ず1文追加する**。①を第一優先で使用：
  - ① このような症状は、多くの人にも見られる比較的よくあるものです。
  - ② 今のような症状は、特別なものではなく、日常的によく見られるケースのひとつです。
  - ③ このような症状は、多くの方に見られる一般的なもののひとつです。
- 本番UIで**絶対に非表示**：抽出症状語、検索クエリ、スコアリング値、ドメイン名、内部JSON、生成プロンプト。
- デバッグ制御：`const IS_DEBUG = false` を既定値にする。

#### ✅ 今すぐやること
- 仕様は `## 10. 「✅ 今すぐやること」の生成ルール` を唯一の正とする。

#### ⏳ 今後の見通し（必須）
目的：
- 一般論や経過説明はしない。
- 「次に迷いが生まれる具体的なタイミング」を1〜2個提示し、その直後に「その時はKairoに戻ればいい」と明示する。

構造（固定）：
1. 状況の自然な流れを一言で述べる（断定しない）
   例：
   ・「このタイプの症状は、時間の経過で変化することがあります」
   ・「しばらく様子を見る中で、気になりやすいタイミングがあります」
2. 「次に迷いやすい具体的トリガー」を箇条書きで1〜2個出す
   ※数値・時間・変化を使い、想像しやすくする
   例：
   ・「もし明日の朝も同じ痛みが続いていたら」
   ・「もし痛みが7以上に強くなったら」
   ・「もし吐き気や発熱が出てきたら」
3. 必ず次の一文で締める（固定表現）
   「そのタイミングで、もう一度Kairoに聞いてください。」

禁止事項：
- 「多くの場合〜」「様子を見ましょう」で終わらせない
- ユーザーに判断を丸投げしない
- 医療的な断定や予測はしない

トーン：
- 落ち着いている
- 先回りして隣にいる感じ
- 不安を増やさず、行動の逃げ道を用意する

#### 🌱 最後に（必須）

**適用範囲**：様子見/市販薬の場合（🟢/🟡）のまとめブロック末尾。5ブロック目の固定セクション。

**目的**：ユーザーを「安心した状態」から「自然に回復行動に入る状態」に導くこと。

**共通ルール**：
- 本文は最大3文まで
- シンプルで短い文章
- 抽象的な励ましは禁止（例：「大切に過ごしてください」）
- 必ず「行動につながる内容」にする
- まとめの最後のブロックとして必ず出力する

---

**🟢/🟡 の場合（様子見・軽症）**

目的：「回復モードに入るスイッチ」を作ること。

必須要素（この順番で構成）：
1. **① 今やるべき行動（最重要）**：必ず「休息」を明確に指示する
2. **② 理由**：回復につながることを説明する
3. **③ 再訪導線（任意だが推奨）**：「また不安になったら、いつでもここで確認してください」等

トーン：優しいが、はっきり指示する。「〜するといいです」より「〜してください」を優先。

**🔴 の場合（受診推奨）**

必須要素：
1. **① 受診の肯定**：「今の状況で受診を選ぶのは適切な判断です」等
2. **② 行動の後押し**：「無理に我慢せず、一度確認してもらうと安心です」等

トーン：不安を煽らない。でも判断ははっきり肯定する。本文は2文以内。

---

**出力形式**：
- 🟢/🟡：`🌱 最後に` + 本文2〜3文
- 🔴：`💬 最後に` + 本文2文以内

**強制ルール**：
- 「🌱 最後に」「💬 最後に」を必ず含める
- セクション単体ではなく、まとめの一部として自然に生成する

**実装**：テンプレート禁止。LLMで会話内容・カテゴリ・主症状に即して毎回生成する。チャット画面のまとめブロック末尾に表示。カテゴリ（PAIN/SKIN/GI/INFECTION）ごとに休息の指示内容を変える。

### 8.2 まとめ後のフォロー質問フェーズ（必須）
- まとめブロック生成後もセッションは継続する（会話終了にしない）。
- **まとめ後フェーズ強制**：まとめを1回出した後は、何があってもまとめを再度出さない。ユーザーが「まとめを出して」「要約して」等と依頼しても、まとめの再出力は禁止。強制的にフォロー質問フェーズのみ返す。
- まとめは初回のみ表示し、2回目以降は絶対に出さない。
- **絶対ルール**: 初回ユーザーメッセージ（1通目）の応答では、サマリーカード・フォロー質問を**絶対に**出さない。必ず次のスロット質問のみを返す。実装で複数層のガードを設ける。
- **初回入力では絶対にtriage禁止**。conversationStep === 0 のときは askFollowUpQuestions のみ返し、triage は実行しない。triage は情報が揃ってからのみ。
- **はじめからボタン**: 完全初期化を実行。部分リセット禁止。`resetConversation()` で conversationStep, triageCompleted, triageLevel, summaryGenerated, collectedAnswers, symptomDuration, painLevel, location, redFlagDetected を全てリセット。サーバー側の conversationHistory / conversationState も削除。クリア前に conversationId を取得してサーバーに送り、該当会話を完全削除してからリロードする。
- まとめ直後に**フォロー質問を1つ**必ず表示する（まとめ内に質問は禁止）。
- フォロー質問は**固定テンプレ**のみ使用する（自由生成禁止）。
- フォローは「判断→行動」を前に進めるための支援に限定する。
- 新しい症状ヒアリング／判断を覆す質問は禁止。
- レイヤーは以下の4層を分離する：ヒアリング → 判断（SNAPSHOT生成） → 表示まとめ → 行動フォロー。

#### Judgment Snapshot（必須）
- 判断確定時に、表示用まとめとは別に `JUDGMENT_SNAPSHOT` を生成して保持する。
- `user_original_phrases` はユーザー発話をそのまま保持する。
- セッション中は破棄しない（表示用まとめを書き換えても SNAPSHOT は保持）。
- フォロー生成は SNAPSHOT を唯一の参照元とする。

```
type JUDGMENT_SNAPSHOT = {
  main_symptom: string
  duration: string
  severity: string
  red_flags: string[]
  risk_factors: string[]
  user_original_phrases: string[]
  judgment_type: "A_HOSPITAL" | "C_WATCHFUL_WAITING"
}
```

#### 判断タイプ
- 🔴 A_HOSPITAL（病院推奨時のみ）
- 🟢 C_WATCHFUL_WAITING（病院推奨以外は必ず🟢）

#### 質問テンプレ（固定）
質問文は固定。回答時の出力内容は**必ず生成**する（`JUDGMENT_SNAPSHOT`、`slotAnswers`、会話履歴を参照）。

1) 質問①（🔴フォロー・診察前準備 or 英語伝え方）
```
今の症状から見ると、念のため病院で確認してもらうと安心そうです。
診察までの間にできることを、
ここで整理しますか？
それとも受診先で英語でどう伝えるか、
一緒に考えますか？
```
出す条件：**A_HOSPITAL（🔴）のみ**

**ユーザーの選択に応じた応答：**
- **「ここで整理」を選んだ場合** → 質問②'の出力（診察までの間にできること、・項目＋→理由、締め）
- **「英語」を選んだ場合** → ①安心文 → ②心理的ハードル下げ → ③【日本語】伝え方 → ④【English】伝え方 → ⑤スマホ一文 → ⑥次の誘い
- **「どっちも」を選んだ場合** → 先に質問②'の出力を表示 → 改行2回 → ④【English】伝え方のみ → ⑤スマホ一文 → ⑥次の誘い
- **どちらでもない場合** → クロージング文

**「英語」を選んだ時の出力（生成必須）：**
- 構成は固定：①安心文 → ②心理的ハードル下げ → ③【日本語】伝え方 → ④【English】伝え方 → ⑤スマホ一文 → ⑥次の誘い
- **③④の内容は必ず生成**：`JUDGMENT_SNAPSHOT`（main_symptom、duration、pain_level等）と会話内容から、ユーザーの症状・経過・つらさを反映した文を生成する。固定例（「2日前から頭痛が…」等）をそのまま使わない。
- 英語には「just to be safe」を必ず含める。「相談」より「診ていただきたい」のニュアンスにする。
- 日本語・英語ともに**必ず文章（完全な文）で出力する**。箇条書きや断片は禁止。

2) 質問②（行動サポート・🟢のみ）
```
今は少し休むだけでも良さそうです。
このまま休みますか？
それとも、もう少し詳しく確認しますか？
```
出す条件：**C_WATCHFUL_WAITING（🟢）** のまとめ直後。

**「休む」などと答えた場合：**
安心＋戻れる一文。内容は生成可（固定文でなくてよい）。

**「詳しく確認」などと答えた場合：**
- 構成：「念のため確認しますね。」→ ☐形式で最大3つ → 「この3つ、できそうですか？」
- **☐の内容は必ず生成**：`JUDGMENT_SNAPSHOT`と「今すぐやること」ブロックを参照し、ユーザーの症状・状態に合った項目を生成する。固定例（「今日は無理をしない」「水分を少しずつとる」等）をそのまま使わない。
- 3つ目の☐は「〇〇が強くなったら気づけるようにする」の型。〇〇は`main_symptom`で埋める。

**☐の質問に「はい」と答えた場合：**
再説明禁止。安心固定のみ。内容は生成可。

**☐の質問に「いいえ」と答えた場合：**
```
どれが難しそうですか？

・（生成：3つの☐に対応する難しさの選択肢）
```
ここで初めて個別対応に入る。例：「水分をとるのがつらい」→「一気に飲まなくて大丈夫です。ひと口ずつでも十分です。」など、生成した☐に応じた個別対応を生成する。

2') 「ここで整理」を選んだ時の出力（🔴のみ・質問①の応答）
**ユーザーが「診察までの間にできることを整理」を選んだと判断された場合の出力（生成必須）：**
- 構成：「診察までの間に、できることを簡単にまとめます。」→ ・項目＋→理由（**最低2件、最大3件**）→ 締め2文。文の間の空行は**2箇所まで**に抑える。
- **・項目と→理由は必ず生成**：`JUDGMENT_SNAPSHOT`と症状（INFECTION等）を参照し、ユーザーに合った診察前の準備を生成する。固定例（「体温を測る」「マスクを用意する」等）をそのまま使わない。
- INFECTION の場合は体温・マスクを優先。それ以外は症状に応じて生成。
- 締め：「今は、病院に向かう準備を優先してください。」「迷ったら、この画面をそのまま見せて大丈夫です。」

#### 回答分岐（🔴質問①の選択判定）
- **「ここで整理」**：整理・まとめ・準備・診察まで・最初・前者・1・一つ目 等
- **「英語」**：英語・伝え方・後者・2・二つ目 等
- **「どっちも」**：両方・どっちも・両方とも・両方お願い 等
- **どちらでもない**：今はいいです／大丈夫です／拒否 等 → **即クロージング**へ。
- クロージングは温かい一言＋判断の尊重＋再利用の余地を必ず含める。
- 「はい」と答えた場合、勝手にクロージングを追加しない。

#### A_HOSPITAL / C_WATCHFUL_WAITING の出力ルール
- 質問文は固定。回答時の内容は**必ず生成**（固定テンプレのコピー禁止）。
- 質問①（🔴）で「英語」を選んだ場合：
  - ① 安心させる ② 心理的ハードルを下げる ③【日本語】伝え方 ④【English】伝え方 ⑤ スマホ一文 ⑥ 次の誘い
  - ③④は`JUDGMENT_SNAPSHOT`と会話内容から**生成**。〇〇病院は不要。「相談」より「診ていただきたい」。「just to be safe」を英語に必ず入れる。
  - **日本語・英語ともに必ず文章（完全な文）で出力する。** 箇条書きや断片は禁止。
- 質問①（🔴）で「どっちも」を選んだ場合：先に診察までの準備を表示 → 改行2回 → ④【English】のみ → ⑤⑥
- 質問②（🟢）：
  - 質問文は固定。「休む」→ 回復モード＋戻れる安心。「詳しく確認」→ ☐形式で最大3つ。**☐の内容は`JUDGMENT_SNAPSHOT`と「今すぐやること」から生成。**
  - 「はい」→ 再説明禁止。安心固定のみ。「いいえ」→ 難しさの選択肢（生成）→個別対応（生成）。
- 質問②'で「はい」（🔴のみ）：
  - **必ず`JUDGMENT_SNAPSHOT`と症状から文を生成する。** 固定例のコピー禁止。
  - 「診察までの間に、できることを簡単にまとめます。」で始める。・項目と→理由は症状に応じて生成（**最低2件、最大3件**）。文の間の空行は**2箇所まで**に抑える。
  - INFECTION の場合は体温・マスクを優先。水分や安静は繰り返さない。命令口調を減らす。
  - 続けて「今は、病院に向かう準備を優先してください。」「迷ったら、この画面をそのまま見せて大丈夫です。」を出す。クロージングは追加しない。

#### C_WATCHFUL_WAITING の特別ルール
- まとめ直後に以下の質問のみ出す：
  - 「今は少し休むだけでも良さそうです。このまま休みますか？それとも、もう少し詳しく確認しますか？」
- 「休む」→ 回復モード＋戻れる安心。「詳しく確認」→ ☐形式（`・`の代わりに☐、最大3つ）。**☐の内容は`JUDGMENT_SNAPSHOT`と「今すぐやること」から生成。**
- ☐に「はい」→ 再説明禁止。安心固定のみ。「いいえ」→ 難しさ選択→個別対応。
- 🟢は「やることを減らす」。様子見＝楽になる。タスク感を出さない。

#### 情報十分性チェック（必須）
- フォロー生成前に `JUDGMENT_SNAPSHOT` の最低要件を確認する（`main_symptom` / `severity` / `duration`）。
- 不足がある場合のみ、行動生成に必要な最小1問だけ確認し、SNAPSHOT の未充足項目を補完する。
- ヒアリングを広げる質問は禁止（最低限の確認のみ）。

## 15. 位置情報と受診先の具体提示（必須）
- UI表示・会話生成・判断ロジックを完全に分離する。
- 位置情報は「LocationSnapshot」として一度だけ確定保存する。
- Snapshot はセッション終了まで上書き・初期化・破棄しない。
- 位置情報が取れなくても会話は止めない。
- 「📍現在地取得済み」は lat/lng のみで判定する。
- ユーザーに現在地をテキスト入力で聞かない。
- 失敗理由をユーザーの責任にしない。

### 15.1 LocationSnapshot（必須）
```
type LocationSnapshot = {
  lat: number;
  lng: number;
  ts?: number;
};
```

### 15.2 正規化関数（必須）
すべての生locationは必ずこの関数を通す。
```
function normalizeLocation(raw: any): LocationSnapshot | null {
  if (!raw) return null;
  if (raw.lat != null && raw.lng != null) {
    return { lat: raw.lat, lng: raw.lng, ts: raw.ts };
  }
  return null;
}
```

### 15.3 conversationState 初期化（必須）
```
function initConversationState(input?: Partial<State>): State {
  return {
    conversationId: input?.conversationId ?? generateId(),
    locationSnapshot: input?.locationSnapshot ?? null,
    clientMeta: input?.clientMeta ?? {},
  };
}
```
- 未初期化代入は禁止。必ず factory 経由で生成する。
- `state.location.xxx = ...` の直接代入は禁止（全体置換のみ）。

### 15.4 Snapshot の確定条件（厳守）
- lat と lng が取得できた時点で snapshot を確定する。
- city / reverse geocode / timestamp は一切条件に含めない。

### 15.5 取得フロー（重要）
- 初回ロード時に自動で取得を開始する。
- lat/lng が取得できた時点で snapshot を確定する。
- snapshot が無い限りは「未取得」扱いを維持する。

### 15.6 UI表示ルール（厳守）
- lat/lng がある場合：
  - 「📍現在地取得済み」を表示する。
- lat/lng がない場合：
  - 「📍現在地を取得できていません」を表示する。

### 15.7 位置情報取得UI（必須）
- 初回のみ以下の文言を必ず一度だけ表示（改変禁止）：
```
より正確な案内のため、現在地を使用できます。
今回は許可しなくても会話は続けられます。
```
- 会話は止めない。
- 初回のみ permission UI を出し、その後は裏で retry。

### 15.8 フォールバック設計
- どの状態でも会話は通常進行。
- snapshot がない場合のみ案内を停止する。

### 15.9 判断ロジック（最重要）
```
function canRecommendSpecificPlace(snapshot: LocationSnapshot | null) {
  return snapshot?.lat != null && snapshot?.lng != null;
}
```
- snapshot がある場合のみ → 具体名OK

### 15.10 医療施設案内（city 非依存・強制案内）
- 施設案内の判断基準は lat / lng のみ。
- snapshot（lat/lng あり）なら必ず案内する。
- city / country / locality は判断ロジックに使わない（表示のみ）。
- 検索範囲は半径 500m〜1km。
- 対象カテゴリ（症状に合わせて選ぶ。GP固定は禁止）：
  - 歯の痛み：歯医者（dentist / dental clinic）
  - 耳・鼻・のど：耳鼻科（ENT / ENT clinic）
  - それ以外：GP（clinic / general practitioner / medical clinic）
  - 重症・緊急が疑われる場合：病院（hospital / medical centre）
  - 薬局（SG）：pharmacy / Watsons / Guardian
- 表示方法：
  - 「近くで行きやすい場所を案内します」を必ず入れる
  - 候補は最大3件、距離が近い順、1件目は「おすすめ」
  - Google Maps をワンタップで開けるリンクを必ず付ける（lat/lng + query）
- city 未取得を理由に案内中止しない
- 「現在地が取得できていないため〜」等の文言は禁止

### 15.10 施設案内の検索ルール（必須）
- 施設案内の判断基準は lat / lng のみ。
- city / country / locality は判断に使用しない（表示用途のみ）。
- 検索半径は 500m〜1km。
- カテゴリ指定：
  - GP：clinic / general practitioner / medical clinic
  - 薬局（シンガポール）：pharmacy / Watsons / Guardian
  - 病院：hospital / medical centre
- 候補は最大3件、距離が近い順。
- 1件目は「おすすめ」と明示する。
- Google Maps のリンクを必ず付ける（ワンタップで開ける）。

#### 注意書き（必須・毎回表示）
以下の固定文のみ表示する：
「迷ったら、今の症状をそのまま薬剤師に見せてください。
一緒に確認してもらえます。」

#### 症状カテゴリ（拡張前提）
- 痛み・発熱
- 喉の違和感
- 鼻水・くしゃみ
- 咳
- 胃の不快感
- 下痢・便秘
- だるさ・脱水気味
- アレルギー症状

### 8.2 病院推奨の場合（A）
必ず以下の4ブロックを順に出す：
1. 📝 今の状態について
2. ✅ 今すぐやること
3. 🏥 受診先の候補
4. 💬 最後に

#### 📝 今の状態について（🔴のみ）
- 箇条書きは `🤝 今の状態について` と同じ構成ルールを使う（最大8行、痛み先頭）。「日常生活では〜」等の前置きは使わない。必ずユーザーの情報を落とさないこと。
- 6スロットの取得内容を表示してよい。
- 「ない / わからない / 不明 / 思い当たらない」等の回答は表示しない。
- ユーザー自由記述は可能な限り原文を保持して反映する。
- 判断は1文で言い切る（なぜ今はGPなのか）。文は固定で、以下のいずれかをランダムで出す。
- 出力例（固定・ランダム選択）：
  - 今の症状の出方をふまえると、念のため医療機関で確認しておくと安心できる状態です。
  - 現在の症状からは、自己判断で様子を見るよりも、一度医療機関で確認しておく方が安心できそうです。

#### ✅ 今すぐやること（🔴のみ）
- ワンクッション文（1文）を最初に表示する。
- 続けて、🟢/🟡と同様の `・<行動>` / `→ <理由>` 形式で出す。
- ③予想経過・④締めの一文は入れない。モーダルも不要。
- **1件目は固定**：
  - `・本日中に医療機関へ連絡する`
  - `→ 早い段階で確認することで、重大な問題でないことが分かるケースも多くあります。`
- **2件目以降（受診までの過ごし方）**は、**モーダル・🟢/🟡と同じクエリ**（10.3：主症状 + 箇条書き全文 + 対処法）を使用して検索し、🟢/🟡と**同じパイプライン**（検索強化・信頼ドメイン優先・フォールバック順など）で生成する。
  - 検索から**必ず2件**を出す（既存の固定項目はこの2件に含めない）。
  - PAIN系・INFECTION系の場合は、以下を**受診までの過ごし方の1件目に必ず入れる**（既存ルール維持）：
    - `・今すぐ受診が難しい場合は、今はベッドに入り、横になって数時間ゆっくり過ごしてください`
    - `→ 体を休息モードに切り替えることで、自然な回復の流れが働きやすくなります。`
  - 上記固定の次に、検索結果から`・<行動>` / `→ <理由>` 形式で**必ず2件**を追加する。

#### 🏥 受診先の候補（施設案内・必須）
- ワンクッションはここでは入れない。
- 🔴（GP推奨レベル）のときのみ、夜間（20:00〜5:59）の場合に限り、最初に次の一文を追加してよい：
- 「現在は夜間の時間帯です。症状が強くなければ、明日受診する形が選択肢の一つです。」
- 夜間以外（6:00〜19:59）は追加しない（朝専用メッセージは禁止）。
- 指示文：位置情報ベース実在医療機関推薦ロジック
  - 目的：
    - 実在する医療機関名を表示する。
    - 緯度経度から半径検索する。
    - 症状に対応可能な施設のみ抽出する。
    - テンプレではない「いいところ」を生成する。
  - 入力条件（必須）：
    - `latitude`
    - `longitude`
    - 主症状
    - 緊急度レベル
  - 医療機関検索ロジック：
    - Google Places API または Maps API を使用する。
    - 検索条件は `type = hospital OR doctor OR clinic`。
    - `radius = 3000m`（都市部）。
    - `keyword = 主症状`（例：`abdominal pain OR gastro OR GP`）。
  - シンガポール専用ロジック（`country === "Singapore"`）：
    - **目的**：シンガポールでは医療受診のほとんどが GP から始まるため、表示する施設を以下の 2種類に限定する。専門科検索（眼科・神経内科など）は使用しない。
    - **国判定**：ユーザーの位置情報（lat / lng）から逆ジオコードで `country = Singapore` を判定する。
    - **検索① 日本人クリニック**：Google Places Text Search クエリ `japanese clinic singapore`
    - **検索② 近くのGP**：Google Places Text Search クエリ `general practitioner`（追加候補：`family clinic`, `gp clinic`）
    - **表示**：2検索を合わせて最大3件
    - **他国**：シンガポール以外は既存の施設検索ロジックをそのまま使用する。
  - 症状適合フィルタ：
    - 腹痛の場合は `gastro / digestive / GP / family medicine` を含む施設を優先する。
    - 整形外科や皮膚科など、症状に無関係な施設は除外する。
  - いいところ生成ルール（`buildHospitalRecommendationReasons`）：
    - **目的**：ユーザーが「ここなら行けそう」と判断できる実用的な情報を提示する。
    - **形式**：箇条書き、最大2つ・最低1つ。短く具体的に。同じ文章を使い回さない。
    - **内容の優先順位**：
      1. 症状との相性（例：皮膚トラブルの初期相談がしやすい、発熱や体調不良の相談で利用されることが多い）
      2. 日本語対応（日本人クリニックの場合）（例：日本語で症状を説明できるため安心）
      3. 通いやすさ（例：現在地から行きやすい場所にあります、駅や主要エリアからアクセスしやすい）
      4. 初診の相談先として使いやすい（例：初期相談の窓口として利用されることが多い）
    - **禁止表現**：説明が丁寧、評判が良い、相談しやすい、人気、おすすめ（根拠が弱くAIらしさが出るため）
    - **出力例**：
      - ① Tokyo Medical Clinic：・日本語で症状を説明できるため安心 / ・発熱など体調不良の初期相談で利用されることが多い
      - ② Orchard GP Clinic：・体調不良の初期相談に対応しているGP / ・現在地から行きやすい場所にあります
  - 表示制限：
    - 病院は2件固定（最低2件、最大2件）。
    - **評価3.8以下の施設は表示しない**。該当する場合は除外し、別の候補を検索・表示する。
    - 表示形式は `① 施設名 ★4.2（12件）` / `② 施設名 ★4.5（8件）`。星評価と口コミ件数を必ず表示する。
    - **施設名は日本語で表示する**（Place Details API で `language=ja` を指定して取得）。
    - 表示内容は `施設名 / いいところ（箇条書き・最低1件、最大2件）`。
    - Google Mapリンクは表示しない（強制）。
  - 禁止事項：
    - 実在しない施設名生成
    - レビューの創作
    - テンプレの「いいところ」
    - 症状に無関係な施設の提示
  - ブランド原則：
    - Kairoは紹介エンジンではない。
    - 「今の状態に合う選択肢を整理する」という立場で出力する。
  - 補足：
    - `INFECTION` カテゴリではオンライン診療案内4行は表示しない（強制）。
    - **🔴のときはオンライン診療案内を表示しない**（強制。MC文削除と同期）。
    - 「行く前に知っておくと安心なこと」セクションは出さない。
    - `city` が取れないことを理由に案内を中止しない。

### 8.1 Places API 実装詳細（現行仕組み）

#### 環境変数
- `GOOGLE_PLACES_API_KEY` または `GOOGLE_MAPS_API_KEY` を指定
- 未設定時は `[]` を返し、検索をスキップ

#### 使用するAPI
1. **Nearby Search** (`/place/nearbysearch/json`)
   - `location`, `radius` または `rankby=distance`, `keyword`, `type`
2. **Text Search** (`/place/textsearch/json`)
   - `query`, `location`, `radius`, `type`（省略可）
3. **Place Details** (`/place/details/json`)
   - `place_id`, `fields`: rating, reviews, types, url, editorial_summary
4. **Geocoding** (`/geocode/json`)
   - 住所→緯度経度、逆ジオコード（緯度経度→都市名）

#### 医療機関検索フロー（`fetchCarePlacesWithFallbacks`）
1. **Nearby Search（半径順）**
   - types: `doctor`, `hospital`, `health`
   - 半径: 3km → 5km → 10km → 20km → 50km（結果が出るまで順に試行）
   - 日本: keywords に `クリニック`, `内科`, `病院`, `医療`, `診療所` を追加
2. **rankByDistance（距離順）**
   - 0件の場合、`rankby=distance` で type ごとに検索（半径制限なし）
3. **Text Search（type 指定あり）**
   - 0件の場合、`type=doctor` で radius 5km〜50km を順に試行
4. **Text Search（type 指定なし）**
   - 0件の場合、`type` なしで検索（より広い結果）
5. **都市名検索**
   - 0件の場合、逆ジオコードで都市を取得し、`[都市名] 病院 クリニック` で検索
6. **0件ログ**
   - キー・位置が有効なのに全戦略で0件の場合、`console.error` でログ出力

#### 候補取得フロー（`resolveCareCandidates`）
1. 位置取得: `locationSnapshot` → ジオコード（住所）→ `getFallbackCoordinates`
2. **国判定**: lat/lng から逆ジオコードで `country` を取得。`country === "Singapore"` の場合はシンガポール専用ロジックへ
3. **シンガポールの場合**（`fetchCarePlacesForSingapore`）:
   - 症状カテゴリ（PAIN/SKIN/GI/INFECTION）に応じてGP検索キーワードを決定
   - **日本人クリニック2件**: Text Search `Japanese clinic` / `Japanese medical clinic` / `Japanese doctor clinic`（ユーザー位置周辺）
   - **GP2件**: カテゴリ別キーワード（PAIN: general practitioner, clinic, medical clinic / SKIN: dermatology clinic, skin clinic, general practitioner / GI: general practitioner, gastroenterology clinic / INFECTION: general practitioner, family clinic）
   - 検索結果が少ない場合は `general practitioner` で補完
   - 並び順: 日本人クリニック → GP、合計4件
   - まとめブロック: 日系1件 + GP1件（おすすめを1件ずつ）
4. **他国の場合**:
   - 症状検出: `detectCareMainSymptomText` で主症状を抽出
   - 検索キーワード: `buildCareSearchQueries` で症状に応じた keywords / includeTerms / excludeTerms を決定
   - `fetchCarePlacesWithFallbacks` で検索
5. `mergePlaces` で重複除去
6. 他国の場合は `applySymptomFitFilter` で症状適合フィルタ（シンガポールはスキップ）
7. `prioritizeCareCandidates` でソート（シンガポールは日系クリニック優先、それ以外は評価・距離順）
8. 上位6件の `placeId` で Place Details を取得し、`rating`, `reviews`, `editorial_summary` を付与
9. シンガポールは最大4件（日本人クリニック2+GP2）、他国は最大2件を返す

#### いいところ生成（`buildHospitalRecommendationReasons`）
症状カテゴリ（PAIN/SKIN/GI/INFECTION）と候補情報に基づき、優先順位に従って最大2件・最低1件を生成する。
1. **優先1 症状との相性**：`getSymptomFitReasonsByCategory` でカテゴリ別の文言を選択（例：PAIN「痛みや体調不良の初期相談で〜」、SKIN「皮膚トラブルの初期相談がしやすい」、GI「腹痛・消化器症状の初期相談で〜」、INFECTION「発熱や体調不良の相談で〜」）
2. **優先2 日本語対応**：`isJapaneseClinicOrSupport` なら「日本語で症状を説明できるため安心」を追加
3. **優先3 通いやすさ**：「現在地から行きやすい場所にあります」「駅や主要エリアからアクセスしやすい」から選択
4. **優先4 初診の相談先**：「初期相談の窓口として利用されることが多い」「体調不良の初期相談に対応しているGP」から選択
5. **重複回避**：`usedReasons` Set で複数候補間で同じ文言を使い回さない
6. **禁止表現**：説明が丁寧、評判が良い、相談しやすい、人気、おすすめ は使用しない

#### 病院・クリニック・薬局検索
- **`resolveHospitalCandidates`**: シンガポールの場合は `[]` を返す（GP・日本人クリニックのみ表示）。他国は type `hospital`, `doctor`, `health`、keywords `病院`（日本）、rankByDistance、Text Search 半径 5km〜50km。位置未取得時はジオコード→`getFallbackCoordinates`
- **`resolveClinicCandidates`**: type `doctor`, `hospital`, `health`、keywords `クリニック`, `内科`（日本）、rankByDistance、Text Search 半径 5km〜20km
- **`resolvePharmacyCandidates`**: type `pharmacy`, `drugstore`、keywords `薬局`, `ドラッグストア`（日本）、Text Search 半径 5km〜50km

#### フォールバック
- フォールバック候補（固定名称）は使用しない
- API 結果が 0 件の場合は `candidates: []` を返し、「位置情報から検索しましたが、見つかりませんでした」と表示

## 9. 「🤝 今の状態について」の出力順
順番を厳守：
1. ユーザーのつらさ・不安への一文の寄り添い
2. ユーザーの事実要約（箇条書き、改行）
3. ユーザーの感覚を言葉にして返す（感覚の翻訳）
4. Kairoとしての判断（断定しすぎない）

追加ルール：
- ③は「今の話を聞く限りだと」など、ユーザーの話が主語になる形にする。
- ③は一般論の説明ではなく、感覚の翻訳を優先する（医者っぽさ禁止）。
- ③で「一般的には〜」など医療説明を主にしない（💊以外で禁止）。
- 診断・確定表現禁止。原因の断定禁止。
- 「注意が必要です」は禁止。代わりに「今すぐ慌てる感じではなさそうです」等を使う。
- 3〜5文程度に収める。
- 「今のあなたの状態なら、こう考えて大丈夫です」「だから今日はこれでいいですよ」は方向性の例。
- テンプレ化はせず、ユーザーを落ち着かせる表現で言い換える。

## 10. 「✅ 今すぐやること」の生成ルール

### 目的
「今すぐやること」は、単なる行動提示ではなく、**納得 → 行動 → 安心**の順で心理的不安を下げる設計にする。
- ユーザーが読んだ後に：今やることが明確／それで良い理由が理解できる／どのくらい続くか想像できる／病院に行かなくてもいい可能性を自然に感じられる、という状態を作る。

### 10.1 出力構造（必ずこの順番）

#### ① なぜそれでいいのか（安心の土台・2文以内）
- 現在の状態は一時的な変化である可能性があるという一般的説明
- 今の段階では刺激を減らすことが合理的である、という理由提示
- 診断・断定禁止。「かもしれません」は最大1回

#### ② 今すぐやること（具体・最大3項目）
- **モーダルと同じクエリ**（10.3：主症状 + 箇条書き全文 + 対処法）を使用して検索し、検索から**必ず2件**を出す（特例の固定項目はこの2件に含めない）。
- **箇条書きの文の長さ・簡潔さ・構成はモーダル（✅ 具体的に）と同一**とする。行動文は簡潔に（長い場合は適宜省略）。理由文も簡潔に。参考表示（参考:...）はモーダルと同様に省略する。
- **主症状・ユーザーの回答を「付随症状」にまとめない**。主症状は主症状として、各スロットの内容を適切に区別して行動・理由に反映する。
- **口調**：勧める口調を使う（`〜してください` / `〜するといいです`）。`・〜します` のような一人称は避ける。
- **PAIN系・INFECTION系で🟡のとき**：1件目を以下で固定する（特例。この固定の次に検索2件を追加）。
  - `・今はベッドに入り、横になって数時間ゆっくり過ごしてください`
  - `→ 体を休息モードに切り替えることで、自然な回復の流れが働きやすくなります。`
- 各件は必ず次の2行セットで出力する。
  - `・<行動>`
  - `→ <理由>`
- 曖昧表現禁止（例：「安静に」だけは禁止）。「何をどのくらい」が分かる具体動作＋軽い理由をセットで出す
- 理由行に `・` は使わない
- 番号付き（`1)` / `1️⃣`）は禁止
- 医療行為の指示は禁止。危険行為・専門処置は禁止

#### ③ 予想経過（安心設計）
- 多くの場合の一般的経過時間を示す（例：「数時間〜1日程度で徐々に落ち着いていくことが多い」）
- 断定禁止。不安を煽る表現禁止

#### ④ 締めの一文（任意）
- 例：「今は体を回復モードに入れることが最優先です。」
- 心理的アンカーとして入れるとよい

### 10.2 表現ルール（厳守）
- **口調**：勧める口調を使う（`〜してください` / `〜するといいです`）。ユーザーにわかりやすく、具体的に伝える。
- 恐怖喚起禁止
- 医学用語の多用禁止
- 「今はこれで安全圏に入る」という安心設計を含意する
- ユーザーの言葉を勝手に変換しない
- ❌「安静にしてください」（抽象的）→ ⭕「体が回復モードに入れる環境を作ってください」
- ❌「水分を取りましょう」→ ⭕「脱水があると痛みが長引くため、少量ずつ補給するといいです」

### 10.3 生成パイプライン（必須）
- `🤝/📝「具体的に」` の生成内容とスロット情報を元に、症状文脈を抽出する。
- **今すぐやること**の検索クエリは、🤝今の状態についてのクエリ（主症状 + 箇条書き全文）に`対処法`を加えて作る（例：`頭痛 痛みは9/10 吐き気がある きっかけは太陽光と睡眠不足 対処法`）。
- **🔴の「受診までの過ごし方」も同じクエリ・パイプライン**を使用する（モーダル・🟢/🟡と同一）。
- **検索の強化**：
  - 日本語・英語それぞれに言語最適化クエリを複数用意し、痛み性状・部位を含むクエリを追加する。
  - 初回検索0件時は、追加クエリ（`主症状 対処法 セルフケア` / `主症状 自宅 ケア` / 英語対応）で再検索する。
  - 全クエリを並列実行し、結果を重複除去・ランク付けする。
- Google Custom Search を日本語・英語で実行する。429時はリトライ、num=10で取得。
- 検索結果は重複除去し、信頼ドメイン（公的機関・大学病院・大手医療情報サイト）を優先する。
- 上位結果の要点を仮説スコアに反映し、行動を決定する。
- 検索結果の生テキストをそのまま表示しない。
- **検索失敗時のフォールバック順**：1) 追加クエリで再検索 2) 検索を使わずLLMで文脈から生成 3) 汎用フォールバックの直前でLLMで再度生成を試行（ここで落ちることはない） 4) 汎用フォールバック（最悪の手段）
- **検索結果を要約して行動を生成する際も、勧める口調（〜してください／〜するといいです）を厳守する。**

### 10.4 仮説選択ルール（実装準拠）
- 特徴量（部位・痛み性状・経過時間・誘因・随伴症状・重症ヒント）で仮説を評価する。
- 主要一致は高配点、禁忌一致は減点する。
- 同点付近では安全側の仮説を優先する。
- OTC推奨は意味判断で決定し、禁忌がある場合は出さない。

### 10.5 理由の確実性（必須）
- 行動文は簡潔に書く（過度な詳細化は禁止）。
- 各理由は、Google検索で得た要点と整合する根拠を必ず含める。
- 理由は一般論だけで終わらせず、「なぜその行動が今の状態に合うか」を明示する。
- このルールは通常のまとめ本文（`✅ 今すぐやること`）と、`✅ 具体的に` モーダルの両方に適用する。

### 10.6 禁止出力（強制）
- 次の行動文は出力禁止：
  - `・症状メモを2時間ごとに1回、合計3回（強さ・変化・随伴症状）で記録し、同日中に悪化サインがないか再確認しましょう`
  - `→ 現在の状態データを再評価しやすくなり、次の判断の精度を維持できます。`

### 10.7 出力制約（必須）
- OTC（ワセリン等を含む市販品）は最大1件。
- OTCを無理に出さない（0件可）。**🟡のときは必ず1件OTCを勧める。**
- `水分を取る / 横になる / 安静にする` は必要に応じて迷わず出す。
- 診断の断定表現は禁止。
- 単なる検索結果の羅列は禁止。
- 参考情報を示す場合は、理由行内で `（参考: <ドメイン> / <ドメイン>）` の形式で示してよい。
- **理由文に検索の痕跡を残さない**：「検索結果より」「検索で」「検索情報では」「上位情報の」などの表現は禁止。理由を substantive に述べる（例：「刺激負荷を減らすことで、症状のぶれを抑えやすくなります」）。

### 10.8 フォールバック強化（必須）
- 検索・LLMが失敗した場合でも、**症状部位（頭・お腹・喉・皮膚）に応じた文脈補足**で埋める。
- **汎用フォールバックの前に必ずLLMで再度生成を試行する**。汎用フォールバックに落ちることはないようにする。
- 汎用フォールバック（「刺激を1つ減らし、水分を…」）は、LLMの再試行が失敗した場合の最終手段のみ。
- 部位別の補足候補を常に用意し、`ensureActionCount` で必ず3件以上を確保する。

## 11. 禁止事項（抜粋）
- 判断の丸投げ質問。
- 行動選択を促す質問。
- 質問フェーズでの助言・判断。
- まとめブロックの欠落。
- 表示上の箇条書きで「-」を使うこと（必ず「・」）。

## 12. 質問生成の内部要件
Kairoが質問を生成する際は、内部的に以下を必ず明確にする。
- この質問で埋める判断スロットは何か
- 直前のユーザー発言から1語以上拾えるか
- 「次に進むため」の目的宣言が入っているか

## 12.1 グローバル禁止（全フェーズ共通）
- 「ない／特にない／該当しない」を不安材料として扱う表現は禁止。
- 否定回答は常に「安心材料／リスク低下情報」として扱う。
- 例：「ないは気になります」「引っかかる」は絶対に使わない。

## 12.2 スロット再質問禁止（必須）
- 各スロットは 1 セッション 1 回のみ質問可能。
- 既に質問済みのスロットは再質問しない。
- 以降は「確認」ではなく「要約・反映」に切り替える。
- 質問導入文（「一点だけ伺います」「ここだけ確認します」等）は未質問スロットにのみ使用可。

## 12.3 まとめ生成ルール（必須）
- slotAnswersをそのまま列挙しない。
- **主症状・ユーザーの回答を「付随症状」として一括して扱わない**。それぞれを適切に区別する。
- 「・ない」「・特にない」単体の箇条書きを作らない。
- 必ず意味のある自然文に再構成する。
- 「日常生活では〜」等の前置きは使わない。取得内容をそのまま記載する。
- ユーザーが「〜です」と答えたものは、「です」を除いて自然な形にする。
- 箇条書き内で一度出たキーワードは繰り返さない。
- 判断・安心コメントは、直前の情報のうち1つ以上を根拠として反映する。

## 12.4 全フロー共通の回答解釈ルール（必須）
### 正規化オブジェクト
ユーザー回答は必ず以下に正規化する。生テキストをそのまま生成に使わない。
```
normalizedAnswer = {
  slotId: string,
  rawAnswer: string,
  riskLevel: "LOW" | "MEDIUM" | "HIGH"
}
```

### riskLevel 割当
- 各スロットの選択肢は事前に riskLevel を割り当てる。
- riskLevel 未定義の回答はエラー扱いとし、生成処理に進まない。

### テンプレート制御
- 文章生成は riskLevel 別テンプレートのみ使用可。
- LOW：安心材料／リスク低下のみ（不安・懸念語は禁止）
- MEDIUM：状況確認／軽い注意まで（断定・強い不安は禁止）
- HIGH：注意喚起／受診検討まで可（恐怖煽り・断定診断は禁止）

### 主観評価語の制限
- 「気になります／引っかかります／心配です／注意が必要です」は HIGH のみ使用可。
- LOW/MEDIUM では絶対に含めない。

### 矛盾防止チェック
- rawAnswer が「ない／特にない」の場合、不安・懸念語が含まれていないこと。
- riskLevel と文調が一致していること。
- 矛盾検出時は該当文を破棄し、同 riskLevel のテンプレで再生成する。

### 適用範囲
- 質問直後のリアクション文
- 状態まとめ（要約）
- 判断コメント
※ いずれも bypass 禁止

## 12.5 初回応答仕様（必須）
- ユーザーが最初に話したとき、安全文を質問の前に必ず表示する。
- 安全文は固定2パターンからランダムで1つ。質問とは分けて出力。安全文が先。
- 〇〇にはユーザーの主症状を短く簡潔にしたもの（頭がいたい→頭痛、唇が痛い→唇の痛みなど）。
- 初回の導入テンプレ（EMPATHY／PROGRESS／FOCUS）は**全て使用禁止**。質問のみ。

安全文テンプレ1:  
「慌てなくて大丈夫です。
多くの〇〇は命に関わるものではありません。
一緒に安全を固めていきましょう。」

安全文テンプレ2:  
「まず安心してください。
〇〇のほとんどは深刻なものではありません。
まず落ち着いて、ひとつずつ安全を確認していきましょう。」

## 12.6 共感テンプレの役割分離（必須）
- 共感テンプレは2種類のみ。
  1) EMPATHY_OPEN（初回のみ）
  2) EMPATHY_NEXT（2問目以降）
- フォールバック用の共感文は禁止。
- templateId / empathyTemplateId が無い場合、共感文は表示しない。
- 共感文はLLM生成禁止。
- questionCount により if/else で明示的に切り替える。
- EMPATHY_NEXT は「今まで通りの共感」を使う（ただし禁止語は除外）。


## 13. 判定設計の基本原則（必須）
- Kairoにおいて「判定完了」と「shouldJudge === true」は同義である。
- 判断スロットが6つ以上埋まった時点、または質問回数が8以上になった時点で、必ず judgeMeta.shouldJudge を true にする。
- **triage_state（必須）**：全APIレスポンスに `triage_state: { triage_level, is_final, required_fields_filled }` を含める。`is_final` が true になるまでサマリー・判定バナーを絶対に描画しない。
- フロントエンドは **triage_state.is_final === true** を唯一のトリガーとして、サマリーカード（🟢🟡🔴・確信度・比率・安心まとめ）を描画する。judgeMeta.shouldJudge は補助のみ。
- サマリーカードの文言は**15文字以内を厳守**。各色ごとに固定文を3つ用意し、ランダムで1つ表示。断定的すぎる医療判断は避け、安心感を優先した言葉にする。
  - 🟢：今は大きな心配なさそうです / 落ち着いて様子を見られそうです / 今のところ安心して過ごせそうです
  - 🟡：今は注意して様子を見てください / 少し注意しながら様子を見ましょう
  - 🔴：早めの受診をおすすめします / 医療機関での確認が必要そうです / 今日中の受診を検討してください
- サマリーカードとまとめブロックの緊急度表記（🟢🟡🔴）は必ず一致させる。矛盾はバグ扱いとする。
- **サマリーカードの色**：6スロット完了時にサーバー内部で算出した緊急度判定（triage_state.triage_level）を唯一の基準とする。クライアントはこれ以外の情報で上書きしない。
- shouldJudge === false の場合、サマリーカードは必ず非表示にする（表示禁止）。
- judgeMeta が存在するだけの状態でサマリーカードを表示してはならない。
- confidence や ratio は「説明用の指標」であり、UI描画の可否を分岐させるためのものではない。
- ⚠️ shouldJudge を false のまま判定完了させてはならない（UIが沈黙するため）。

## 14. 判断スロット設計（必須）
Kairoは以下6つの「判断スロット」を持つ。質問の有無に関係なく、ユーザー発話または質問応答で値が取得できた時点で「埋まった」と判定する。
1. 痛みの強さ
2. 悪化傾向
3. 経過時間
4. 日常生活への影響
5. 付随症状
6. 想定される原因カテゴリ（本人の認識）

### 14.0 固定順・固定文言（必須）
質問の形式は以下を維持する。未充填スロットのみ、次の固定順で1ターン1問ずつ出す。
（filled のスロットはスキップして次へ進む。6問必ず出す構造は禁止）
1. 痛みの強さ：  
　「今の痛みを、1から10であらわすと何点ですか？」
2. 悪化傾向（痛み方）：  
　「今の痛み方はどれに近いですか？」  
　・（AIが生成：症状に合った痛み方）  
　・（AIが生成：症状に合った痛み方）  
　・（AIが生成：症状に合った痛み方）
3. 経過時間：  
　「いつから始まりましたか」  
　・さっき  
　・数時間前  
　・一日前
3.5 悪化傾向（特例）：  
　「さっき」以外の回答（「数時間前」「半日」「昨日」「三日前」「一日以上前」など）が来た場合、3と4の間に挿入する。全カテゴリ共通。  
　「今の方向性はどうですか？」  
　・回復に向かっている  
　・変わらない  
　・発症時より悪化している  
　※「発症時より悪化している」選択時、かつ痛みの強さ（1問目）が5点以上の場合のみ必ず🔴とする。まとめにも出すこと。
4. 日常生活への影響：  
　「今の動ける感じはどれに近いですか？」  
　・普通に動ける  
　・少しつらいが動ける  
　・動けないほどつらい
5. 付随症状（PAIN・INFECTION の4問目）：  
　「これ以外の症状は他にありますか？」  
　・これ以外は特にない  
　・吐き気がある  
　・咳や発熱がある（ユーザーが先に「頭が熱い」等と言っていた場合は・咳があるのみ）
6. 想定される原因カテゴリ（本人の認識）：  
　「何かきっかけで思い当たることはありますか？」  
　・スマホやパソコンを長時間見た  
　・寝不足や疲れが続いている  
　・強いストレスや緊張があった

上記3択に当てはまらない自由記述は、従来の「自由記述回答の扱い（固定）」を適用する。
まとめでは「自由記述回答の扱い（固定）」を適用し、ユーザー原文を優先して記載する。

### 14.0 ユーザー回答の扱い（必須）
- **緊急度判定時以外**：全てのユーザー回答をそのまま、フィルタを通さずに使用する（まとめ・クエリ・会話履歴など）。
- **緊急度判定時のみ**：一番近い選択肢に合わせた数値化が必要。それ以外の用途では一切フィルタを通さない。
- **出力の型整形**：情報整理・まとめでは、ユーザーの自由記述をそのまま出さず、型に合わせて整形する（例：三日前→「三日前に症状が始まった」）。マッピングは緊急度判定の計算時のみ行う。
- **LLM出力の簡潔化**：「〜の可能性があります」→「〜の可能性」のように簡潔にするフィルタを適用する。

### 14.1 質問制御ルール
- 毎ターンの処理順序は以下で固定する：  
  `normalize_text() -> detect_symptom_type() -> extract_slot_values() -> update_slot_status() -> generate_next_action()`
- 毎ターン、ユーザー発話の解析→自動充填を先に実行する。
- 各質問は必ず、未充填スロットを1つ埋める目的を持つ。
- filled == true のスロットに質問することを禁止する。
- 同一スロットに2回質問することを禁止する。
- 1ターンで2問以上出すことを禁止する。
- 6スロットが全て埋まった時点で判定・まとめへ移行する。

### 14.1.1 正規化（必須）
- 全角数字は半角数字に正規化する。
- 「4点」「4くらい」「4/10」などは痛みスコア抽出時に `4` として扱う。
- 「きのう」は「昨日」に正規化する。
- 「かも」「と思う」等の曖昧語は削除しない（原文保持）。

### 14.1.2 抽出ルール（必須）
- severity：`1-10`、`◯点`、`◯/10` を抽出。
- worsening：`worsening/improving/stable/fluctuating` の trend と、痛み方 quality を分離して保持。
- duration：`raw_text` と `normalized` を保持し、表示は `raw_text` を使う。
- impact：生活機能への影響表現から抽出。
- associated：主症状以外を配列で保持。
- cause_category：原因推定語と不確実性表現（「かも」「と思う」）を保持。

### 14.1.3 上書きルール（必須）
- 既に filled のスロットは原則上書きしない。
- 明確な訂正意図（例：「やっぱ」「訂正」「言い直し」）がある場合のみ上書きを許可する。

### 14.2 slot_status（必須）
```
slot_status = {
  severity: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  worsening: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  duration: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  impact: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  associated: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null },
  cause_category: { filled: boolean, value: string | null, source: "user_spontaneous" | "question_response" | null }
}
```
- value にはユーザー原文を優先して保持する。
- 既に filled のスロットは上書きしない（明示的訂正時を除く）。

### 14.4 導入文テンプレ制御方式（必須）
導入文は **役割×バリエーション** のテンプレID制のみで出力する。  
LLMの自由生成は禁止。

#### 役割（role）
- EMPATHY（受け止め）
- PROGRESS（小さな前進）
- FOCUS（今見るポイント）

#### ルール
- 質問ブロックは「導入文（最大2文）→ 質問文 → 選択肢」の順。
- 各役割は3〜5個の文言バリエーションを持つ。
- 同一テンプレIDはセッション内で再利用禁止。

#### 質問スロット × 導入テンプレ対応表（必須）
1) 初回（痛みスコア）  
  - 安全文を質問の前に必ず出す（固定2パターンからランダム）。質問とは分けて出力。安全文が先。
    - 「慌てなくて大丈夫です。
    多くの〇〇は命に関わるものではありません。
    一緒に安全を固めていきましょう。」
    - 「まず安心してください。
    〇〇のほとんどは深刻なものではありません。
    まず落ち着いて、ひとつずつ安全を確認していきましょう。」
  - 〇〇にはユーザーの主症状を短く簡潔にしたもの（頭がいたい→頭痛、唇が痛い→唇の痛みなど）
  - 導入テンプレ（EMPATHY／PROGRESS／FOCUS）は**全て使用禁止**。質問のみ。  

2) 経過時間  
  - 使用可：FOCUS  
  - 禁止：EMPATHY／PROGRESS  

3) 悪化傾向  
  - 使用可：FOCUS  
  - 禁止：EMPATHY／PROGRESS  

4) 日常生活への影響  
  - 使用可：PROGRESS（セッション中最大1回）／FOCUS  
  - 禁止：EMPATHY  

5) 付随症状  
  - 使用可：FOCUS  
  - 禁止：EMPATHY／PROGRESS（既に使用済みなら不可）  

6) 原因・きっかけ  
  - 使用可：PROGRESS（未使用なら可）／FOCUS  

#### 最後の質問（必須）
- **最後の質問では何があっても**、PROGRESS／FOCUS を禁止。**質問文のみ**にする。

#### 再利用禁止ルール（必須）
- 共感／前進／目的の言い回しは直近2問と同じ表現を避ける。
- ①②③は直近5ターン以内の文・構文・語順・語尾・リズムの再利用を禁止。
- 同じ導入語・結びが続く場合は必ず言い換える。

#### 初回例外
- 初回は安全文＋質問のみ。導入テンプレ（EMPATHY／PROGRESS／FOCUS）は使用しない。
#### 追加ルール
- 1〜3問目は②小さな前進の言語化を必ず省略する。

### 14.5 質問導入文のテンプレート制御（必須）
導入文（①共感／②前進／③目的宣言）は、LLMの自由生成から完全に切り離す。
LLMは**テンプレIDのみ**を返し、文言はフロントで静的テンプレから挿入する。

#### LLM出力形式（固定）
```
{ "templateId": "FOLLOW_UP_SHORT", "question": "質問本文" }
```

#### 文タイプ（role）
- EMPATHY
- PROGRESS_CONFIRM
- NEXT_STEP_DECLARATION

#### テンプレート制御ルール
- 初回質問：安全文＋質問のみ（導入テンプレ不使用）
- 2問目以降：PROGRESS_CONFIRM は最大1回まで
- 同一テンプレIDは session 内で再利用禁止
- 選択済みテンプレIDは state で管理し、LLMには渡さない

#### 追加ルール
- 経過時間は数値で聞かない（◯時間・◯日は禁止）
- 「短期 or 持続」が分かれば十分
- ユーザーに思い出させたり考えさせる聞き方は禁止
- 共感文はユーザーの発言を自然な文に整形してから使う（「〜ですは」等の不自然文は禁止）

### 14.2 まとめとサマリー表示
- まとめ生成後も会話は継続する（FOLLOW_UP_STATE）。
- サマリーカードは、まとめブロックの先頭（🟢 ここまでの情報を整理します または 📝 今の状態について）を出すタイミングと**同時に**表示する。
- 判定後の表示順は必ず以下の順とする：
  1. サマリーカード
  2. 🟢🟡🔴の判定（まとめブロック先頭の見出しで表現）
  3. まとめブロック

### 14.3 質問生成ルール（追加）
- 判断スロットは固定だが、質問文は固定してはいけない。
- 質問文は直前のユーザー発言・感情・文脈を必ず反映して生成する。
- 同じ判断スロットでも、会話の流れによって質問の聞き方を変える。
- テンプレ質問を連続して使用することは禁止。
